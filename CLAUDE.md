# Botkin — инструкции проекта

Медицинская система распознавания (OCR/VLM) лабораторных результатов и рецептов.
Стек: Python, TDD (pytest), ruff, Ollama (qwen3-vl), pymupdf, rapidfuzz, реестры ГРЛС/ФСЛИ.

## Протоколирование для Хабра и доклада на конференции — ОБЯЗАТЕЛЬНО

Этот проект ведёт публичный технический дневник. **Любую находку фиксируй в журнале
`habr/lab-results-journal.md` со всеми данными, пригодными для статьи и доклада.** Это не
опционально и не «по запросу» — делай это всегда, как часть завершения работы.

Что протоколировать (со всеми деталями, не пересказом):
- **Баги и их корневые причины** — что наблюдалось, как воспроизвели, какой диагноз
  оказался неверным и почему, какой корень нашли.
- **Инсайты и неочевидные выводы** — почему проблема была именно во входе/данных/промпте.
- **Выявленные проблемы и ограничения** — даже если не чинили в этой итерации.
- **Решения** — какой фикс, под каким тестом (TDD: RED → GREEN), почему именно так.
- **Конкретные данные** — реальные значения, единицы, имена показателей, числа тестов
  (`N passed`), команды воспроизведения, фрагменты вывода. Без обобщений «починили баг».

Формат — продолжай нумерацию `## Итерация N: <заголовок>` с разделами
**Проблема → Диагноз → Решение → Итог**. Доказательства (evidence) приводи дословно:
сравнения «до/после», вывод инструментирования, отличия рабочего и сломанного случая.

Не выдумывай числа и цитаты — только то, что реально наблюдалось в прогоне (см. Stop-list
в глобальных инструкциях).

## Рабочие команды

- Тесты: `.venv/bin/python -m pytest`
- Линт: `.venv/bin/ruff check src tests`
- Push: токеном из `token.txt` (x-access-token); `gh` CLI недоступен — PR через GitHub API.
