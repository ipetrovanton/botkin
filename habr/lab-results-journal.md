# Журнал реализации: распознавание лабораторных анализов

> Фактура для статьи на Хабре и доклада на конференции. Пишется по ходу работы:
> решения, находки, грабли, обоснования порогов. Не причёсано — сырьё для нарратива.

## Контекст задачи

Бот распознаёт медицинские документы (фото/PDF) через локальную VLM (qwen3-vl + Ollama,
instructor для structured output). Лабораторные анализы **терялись** в pipeline —
трёхслойный баг: извлечение (падение валидации), классификация, рендер карточки.

**Цель блока:** анализы надёжно проходят pipeline и выводятся в Telegram; распознанные
названия нормализуются по справочнику **ФСЛИ** (Федеральный справочник лабораторных
исследований, OID 1.2.643.5.1.13.13.11.1080, портал НСИ Минздрава) — каноничное имя +
LOINC + код НМУ; единицы измерения сверяются с эталонными.

**Метод:** строгий TDD (red-green-refactor), 11 задач. Архитектура повторяет уже
проверенный блок B (нормализация препаратов по ГРЛС): офлайн-скрипт `xlsx → registry.jsonl`,
ленивый синглтон-нормализатор на `rapidfuzz`, декларативная миграция БД.

## Task 5: скрипт сборки реестра ФСЛИ (xlsx → jsonl)

**TDD-цикл:**
- RED: `tests/test_build_analyte_reference.py` (5 тестов на чистых функциях
  `row_to_record` / `normalize_key` / `STATUS_MAP`) → `ModuleNotFoundError` (модуля нет). ✓
- GREEN: `scripts/build_analyte_reference.py` — 5/5 passed.
- REFACTOR: добавил guard `if ws is None` (как в `build_drug_reference.py`); ruff чисто.

**Структура источника ФСЛИ (xlsx):** один лист «Справочник», шапка в строке 2 (1-based),
данные с 3-й. ~18 значимых колонок; берём LOINC, FULLNAME, ENGLISHNAME, SHORTNAME,
SYNONYMS (разделитель `;`), UNIT, GROUP, TESTSTATUS, NMU.

**Дизайн-решения (фактура для статьи):**
- **Параллель с блоком препаратов.** Скрипт сознательно повторяет структуру
  `build_drug_reference.py` (ГРЛС): те же `normalize_key`, `_meta` первой строкой jsonl,
  `write_registry`, `argparse` + `# pragma: no cover` на `main`. Единый паттерн «офлайн-сборка
  справочника из госвыгрузки → jsonl» переиспользуется между регистрами Минздрава.
  Разница: ГРЛС — ZIP из 8 листов-статусов, ФСЛИ — один xlsx-лист.
- **`normalize_key`: ё→е + lower + схлопывание пробелов.** OCR и сами бланки путают «ё/е»;
  ключ дедупликации и матчинга должен быть к этому устойчив. Тест-кейс `«  Гёмоглобин  Общий »`
  → `«гемоглобин общий»` фиксирует все три преобразования разом.
- **LOINC «0» → null.** В выгрузке отсутствие кода LOINC закодировано строкой "0", а не пустотой.
  Если не вычистить — в БД попадёт мусорный код «0», ломающий связку с LOINC. Отдельный тест.
- **Дедупликация по нормализованному имени.** Первый победитель остаётся (`seen`-множество).
- **`STATUS_MAP`: Актуальный/Новый/Устаревший → active/new/deprecated.** Статус теста несём
  до рантайма, чтобы потом подсвечивать устаревшие позиции в карточке.

Коммит: `feat(reference): скрипт сборки справочника анализов из ФСЛИ (xlsx → jsonl)`.

## Task 6: генерация registry.jsonl — две засады в реальном файле

Первый прогон скрипта на живом `Справочник лабораторных тестов.xlsx` дал **«Записано 0 тестов»**
(ожидалось ~20727). Диагностика вскрыла два расхождения плана с реальностью — обе попадут в статью
как иллюстрация «офлайн-данные госреестров полны сюрпризов, читай файл, не доверяй спеке».

**Засада 1 — битые `dimensions` + `read_only=True`.**
В `read_only` режиме openpyxl берёт размеры листа из сохранённых метаданных, а выгрузка ФСЛИ
записала туда `A1:A1` (хотя реально `A1:R20729`). Итог: `iter_rows` отдаёт ровно одну ячейку
`'ID'` и обрывается → ноль данных, без ошибки. Тихий провал, как и `extra="forbid"` ранее.
Перепроверка с `read_only=False` показала реальные 20729×18.
Фикс: `ws.reset_dimensions = True` (заставляет openpyxl вычислять границы по мере чтения,
сохраняя потоковый режим и не вытягивая 20k строк в память).

**Засада 2 — двойная шапка.** План предполагал шапку в строке 2. Реально:
- строка 1 — **машинные** имена колонок: `ID, LOINC, FULLNAME, ENGLISHNAME, SHORTNAME, SYNONYMS, …`
- строка 2 — **человекочитаемые** описания: «Уникальный идентификатор», «Код LOINC», «Полное наименование», …
- строка 3+ — данные (первая запись: АЧТВ, LOINC 14979-9, НМУ A12.05.039, ед. «с»).

`row_to_record` матчит по машинным ключам (`FULLNAME`, `LOINC`), значит шапка — строка **1**,
а строку 2 (описания) надо явно пропустить — иначе она прошла бы как валидная запись с именем
«Полное наименование». `_HEADER_ROW=1`, `_DATA_START_ROW=3`.

**TDD-замечание:** баг `read_only`/dimensions специфичен для конкретного битого файла и не
воспроизводится синтетическим xlsx (openpyxl при сохранении пишет корректные dims). Поэтому
логику парсинга (шапка=стр.1, пропуск стр.2, данные=стр.3+) покрываю unit-тестом на временном
xlsx, а фикс dimensions верифицирую реальным прогоном (число записей > 0).

**Развязка засады 1.** Первая попытка фикса — `ws.reset_dimensions = True` в read-only режиме —
**не сработала** (прогон снова дал 0; прямая проверка: `iter_rows` всё ещё отдаёт 1 строку).
В потоковом read-only парсере openpyxl `reset_dimensions` не переопределяет границы из битого
`<dimension>`. Рабочее решение — `read_only=False` (полная загрузка вычисляет реальный
`A1:R20729`). Для разового офлайн-скрипта держать ~20k строк в памяти приемлемо. Урок для статьи:
«ленивый стриминг экономит память, но доверяет метаданным файла — а госвыгрузка им врёт».

**Результат сборки:** 20720 уникальных тестов (из 20727 строк; 7 — дубли по нормализованному имени).
Спот-чек качества:
- Гемоглобин → LOINC `718-7`, НМУ `A09.05.003`, ед. «г/л» ✓
- АЧТВ → LOINC `14979-9`, НМУ `A12.05.039`, ед. «с» ✓
- Статусы: **20579 active / 93 deprecated / 48 new**.
- **Заполненность полей: LOINC — 31%, НМУ — 37%, единица — 78%.**

**Находка для калибровки и статьи:** LOINC проставлен лишь у ~трети позиций ФСЛИ, НМУ — чуть
больше трети. То есть «обогащение анализа кодом LOINC» сработает не всегда — в карточке это надо
показывать как опциональное поле, а не обещать всегда. Единица измерения есть у 78% — на неё
можно опираться для сверки (блок проверки единиц, Task 9). Синонимов много и они разноязычные
(`["Активированное частичное…", "APPT", "aPTT, aPTT PPP, PTT"]`) — хорошая база для фаззи-матчинга.

## Task 7: пороги нормализатора в config

Тривиальный TDD-цикл. Добавлены `ANALYTE_MAX_EDIT_RATIO=0.35` и `ANALYTE_RATIO_FLOOR=75`.
**Замечание:** пороги для анализов строже, чем для препаратов (`0.40`/`70`). Обоснование —

## Task 8: AnalyteNormalizer — фаззи-коррекция по ФСЛИ

**TDD:** 9 тестов (8 на синтетике + 1 на упакованном реестре 20720 записей) → все зелёные.

**Дизайн-решения (фактура):**
- **Scorer = абсолютная дистанция Дамерау-Левенштейна** (как в блоке препаратов). Выбран потому,
  что OCR-ошибки — это расстояние 1–3 (з→э, и→н), и distance-scorer ставит верный ответ первым,
  тогда как WRatio/JaroWinkler на большом словаре дают ложные совпадения. `cap = max(1, floor(len ×
  0.35))` — короткие имена строже. Плюс пол по `fuzz.ratio ≥ 75`.
- **Разворот записи в множество поисковых ключей.** Каждый тест индексируется по полному имени,
  краткой форме, английскому имени и всем синонимам → одна каноничная запись. Так «CRP», «СРБ» и
  «С-реактивный белок» все ведут к одной записи. Первый победитель за ключ остаётся (детерминизм).
- **Короткие ключи (≤3 символов) — ТОЛЬКО точное совпадение.** Ключевое решение: на аббревиатурах
  «HGB», «СРБ», «АЛТ» фаззи с дистанцией 1 склеивал бы разные показатели (3 буквы, расстояние 1 =
  каждая вторая аббревиатура «похожа»). Поэтому для коротких запросов — строгий lookup по словарю,
  иначе `unverified`. Тест `test_short_abbreviation_requires_exact`: «HGB»→Гемоглобин, «XYZ»→unverified.
- **Принцип безопасности (как у препаратов): не подменять при сомнении.** Если ни порог по
  расстоянию, ни ratio-floor не пройдены — имя НЕ заменяется, оригинал сохраняется в `raw`,
  `status='unverified'`. Защищает редкие/новые показатели от ложного «исправления» в чужой анализ.
- **`status` теста несётся в `match_status`** (active/new/deprecated) — чтобы карточка могла
  пометить устаревшую позицию ФСЛИ.

единица «г/л») → RED (нет синглтона) → GREEN.

**Решения:**

- **Побочно починены `ref_operator`/`ref_text`:** старый `_persist_lab` их не писал в БД (хотя

## Post-Task 11: смена источника сборки xlsx → JSON

Пользователь принёс ту же выгрузку ФСЛИ в JSON (`{"records": [...]}`, 20727 записей, ключи —
машинные имена колонок). Решение: **рантайм не трогаем** — `AnalyteNormalizer` в проде читает
не входную выгрузку, а собранный `registry.jsonl` (компактный, по строке на запись, ленивая
загрузка). JSON меняет только офлайн-шаг сборки.

**Что дал переход (по TDD: RED на синонимах и JSON-чтении → GREEN):**
- **Убраны засады xlsx.** Чтение через `json.loads` вместо `openpyxl` — больше нет битого
  `<dimension>` (требовавшего `read_only=False`) и двойной шапки (строка описаний). Плоский
  JSON разбирается стандартной библиотекой, детерминированно. Тема для статьи: *«тот же
  госреестр в JSON оказался надёжнее xlsx — формат выгрузки как источник хрупкости»*.
- **Починен баг разделителей синонимов.** ФСЛИ смешивает `;` и `,` в одном поле
  (`"...; APPT; aPTT, aPTT PPP, PTT"` — 771 запись). Старый split только по `;` склеивал
  хвост в один мусорный «синоним». Перешли на `re.split(r"[;,]")`: **2438 записей получили
  больше форм, +4561 синоним всего** (11177 → 15738) — это прямой прирост покрытия фаззи-матчинга.
- **Результат сборки идентичен по составу:** те же 20720 нормализованных ключей 1:1, изменения
  только в синонимах. Зависимость `openpyxl` оставлена — её ещё использует `build_drug_reference.py`
  (выгрузка ГРЛС приходит .xlsx внутри zip).

**Грабля-омоним:** антигены системы Резус `E` и `e` — два разных антигена, но `lower()` в ключе
дедупа схлопывает их в один; реестр сохраняет лишь одного «победителя» (зависит от порядка строк
в выгрузке). Дефект дедупа давний, не от смены источника, но всплыл при diff'е. Урок: регистр-
зависимые сущности в медицине существуют, агрессивная нормализация ключа может терять данные.

Итог: `uv run pytest -q` → **146 passed**, ruff чисто.

## Калибровка на живой Ollama: настоящий корень «пустого извлечения»

Прогон `sample_020.pdf` (Инвитро, 2 стр.) на живой qwen3-vl:8b показал: документ
классифицируется верно (analysis, title/clinic извлечены), но `lab_results` пуст,
`raw_extraction="[]"`. Карточка рендерит шапку и «—».

**Сначала добавили наблюдаемость** (без неё — гадание):
- Логирование в API-процессе вообще не было настроено (`basicConfig` жил только в боте),
  а pipeline крутится именно в API → `[*_EXTRACT]` в консоль не выводились. Вынесли
  `setup_logging()` (LOG_LEVEL) в общий модуль, подключили в обоих entrypoint.
- Залогировали границы: число страниц PDF и размеры рендера; число изображений и объём
  base64, уходящих в VLM; число распознанных строк; при нуле — **сырой ответ модели**
  до парсинга (`[EMPTY_EXTRACT]`).

**Что показал лог (опровергнув две гипотезы по дороге):**
- Не препроцессинг: PDF→2 изображения 1555×2200, ~1.2 МБ base64 — норма.
- Не переполнение контекста: `Prompt: 4096 t` при `num_ctx=16384` — с запасом.
- Сырой ответ — **валидный JSON со всем ОАК**, но в СВОЕЙ схеме:
  `{patient_id, ..., tests:[{test_name, results:[{parameter, value, unit, reference_range, comment}]}]}`.

**Корень — тихий рассинхрон схем.** instructor ждал `LabResults{results:[LabResult]}`
(плоско, поля `analyte_name`/`value_num`/`ref_low`). На верхнем уровне ответа ключа
`results` нет → pydantic берёт дефолт `results=[]`, лишние поля игнорируются (`extra=ignore`,
снятый в Task 3 ради устойчивости — здесь он же замаскировал проблему!) → валидация
проходит без ошибки, retry не срабатывает. Классический тихий отказ: «успех» + ноль данных.

**Фикс — адаптер под вывод модели** (вместо борьбы промптом с 8B-моделью, которая уже
проигнорировала инструкции): pydantic-схема `RawAnalysis` под фактическую вложенную
структуру + маппер `rows_from_raw` в плоский `LabResult`. Парсинг значения и нормы
перенесён из промпта в КОД (надёжнее модели): `parse_lab_value` («44.6*»→44.6, текст→
value_text), `parse_reference_range` («35 - 45»→low/high, «< 1.0»→оператор, «≤/≥»→</>,
текст→ref_text). instructor-схема теперь совпадает с естественным выводом модели — рассинхрон
снят в корне; промпт переписан под ту же структуру. Маппер проверен на реальном сыром
ответе из лога: 6/6 строк ОАК с верными значениями, единицами, нормами и флагами.

Тема для статьи: **«`extra=ignore` спасает от падений извлечения и одновременно прячет
рассинхрон схемы — наблюдаемость (сырой ответ модели в логе) важнее, чем кажется»**; и
**«с локальной 8B-VLM дешевле принять её естественный JSON и распарсить в коде, чем
заставлять следовать вашей схеме»**.

Итог: `uv run pytest -q` → **164 passed**, ruff чисто. Финальный E2E через Ollama — за
пользователем (среда разработки без GPU), логика маппинга доказана на фактическом ответе модели.

## Итерация 2: модель недетерминирована в схеме — и теряет страницы

Прогон с адаптером под именованные поля снова дал `[]`. Сырой ответ показал, почему:
**qwen3-vl:8b каждый прогон раскладывает данные по РАЗНЫМ ключам**. Прогон №1 —
английские (`tests/results/parameter/value/unit/reference_range`); прогон №2 — русские
(`Результат`/`Исследование`/`Единицы`/`Референс`) и верхний ключ — пустая строка `""`.
Данные по содержанию верные каждый раз, но имена ключей выдуманы заново. Перечислить их
все в схеме/алиасах невозможно.

**Вывод:** парсить надо по СОДЕРЖИМОМУ значений, не по именам ключей. Гибрид:
1) структурный разбор по именованным полям (быстрый путь, покрывает «послушные» ответы);
2) fallback — `harvester`: рекурсивно ищем в любом JSON массивы строк-показателей и
классифицируем поля по виду значения (число / диапазон «35 - 45» / оператор «< 1.0» /
единица) с подсказкой по алиасам ключей. Имена ключей не важны. Проверено на ОБОИХ
реальных ответах (англ. и рус.) — один и тот же результат.

**Вторая проблема, всплывшая на реальном PDF:** `sample_020.pdf` — два отдельных бланка
(стр.1 — С-реактивный белок, стр.2 — клинический анализ крови). При двух изображениях в
одном запросе модель вернула данные ТОЛЬКО со стр.2 — первая страница «потерялась». При
этом `Prompt: 4096 t` при `num_ctx=16384` — контекст НЕ переполнен; увеличивать его
бессмысленно (и медленнее). Решение — **гибридная многостраничность**: один общий вызов;
если распознанных исследований меньше, чем страниц, — добор каждой страницы отдельным
вызовом и объединение с дедупом. Платим лишними вызовами только когда модель реально
потеряла страницу.

**Заголовок документа.** «С-реактивный белок» как название всего документа — плохо.
Обобщаем по преобладающему биоматериалу (`specimen` ФСЛИ, заполнен на 100%):
Кровь/Сыворотка/Плазма → «Анализ крови», Моча → «Анализ мочи». Fallback — название
исследования из бланка → «Лабораторные анализы».

Темы для статьи: **«structured output на локальной 8B — не контракт, а лотерея имён
ключей; парсьте инвариант (содержимое значений), а не синтаксис»**; **«VLM при нескольких
изображениях молча теряет страницы — детектируй по покрытию и добирай»**; **«полнота
контекста ≠ его размер: Prompt 4096/16384 опроверг гипотезу переполнения за секунду —
если есть лог»**.

Итог итерации 2: **180 passed**, ruff чисто. Осталась калибровка скорости (`extract_long_side`)
на живой Ollama — за пользователем.

## Итерация 3: pipeline отработал, а UI завис — отладка многопроцессной доставки

Первый живой E2E через Telegram дал неожиданное: pipeline распознал документ полностью,
а бот навсегда завис на «🧩 Нурмализую данные». Классический случай, где «успех» в одном
процессе не равен «результат у пользователя».

### Сложность: бот зависает на «Нормализую данные», результат не приходит

- **Симптом:** прогресс-бар в Telegram застывает на стадии «🧩 Нормализую данные ●» и
  не превращается в карточку. При этом логи API-процесса показывают полный успех:

  ```
  [SUCCESS_EXTRACT] ... Распознано строк: 0          # общий вызов — пусто
  [MULTIPAGE] ... добор постранично
  [SUCCESS_EXTRACT] ...#стр2 ... Elapsed: 79.47s      # одна страница — 79 секунд!
  [EXTRACT_QUALITY] строк: 27 | с единицей: 27
  [NORMALIZE_QUALITY] сопоставлено ФСЛИ: 16/27
  Doc 18: заголовок обобщён → 'Анализ крови'
  Doc 18 processed                                     # статус extracted записан в БД
  ```

- **Гипотезы (три, конкурирующие):**
  - **D1** — исключение в фоновой задаче бота `asyncio.create_task(run_progress_flow(...))`:
    задача без `await` и без `done`-callback глотает любое исключение молча → поллинг
    обрывается, сообщение застывает на последнем рендере.
  - **D2** — гонка доставки: бот ловит `extracted`, но `claim_delivery()` возвращает
    `False` (push-fallback pipeline забрал доставку первым). Ветка `if claimed:` БЕЗ
    `else` → сообщение не обновляется, результат уходит отдельным «/show»-уведомлением.
    Зависит от `DELIVERY_FALLBACK_DELAY` (env, дефолт 130 с).
  - **D3** — таймаут поллинга бота (120 с) ≈ время обработки (~108 с в этом прогоне,
    раздутое 79-секундным добором стр.2). Чуть медленнее — `poll_until_done` вернёт `None`.

- **Что пробовал (по методу — сначала исключить, потом чинить):**
  - Проверил, видит ли бот свежий статус. `get_conn()` открывает **новое** соединение
    SQLite на каждый poll (`isolation_level=None`, закрывается в `finally`) → коммиты
    API-процесса видны сразу. Гипотеза «бот читает устаревший статус» — **отпала**.
  - Проверил рендер карточки. `_format_labs`/`format_card_header` обращаются к полям
    через `.get()` — на данных Doc 18 (значения вида «44.6\*», 11 нераспознанных) не
    бросают. Гипотеза «падает форматирование» — **ослаблена**.
  - Полез в логи бота — и нашёл, **почему их нет**: логгер модуля назывался
    `logging.getLogger("bot.upload")` — вне дерева `botkin`. А `setup_logging()`
    конфигурирует только логгер `botkin` (`logger.propagate = False`). Итог: весь путь
    доставки писал в ненастроенный логгер → **полная немота**. Дважды слепы: и кода
    логирования не было, и тот, что был, не выводился.

- **Решение (этой итерации — инструментирование, не фикс вслепую):** по методике
  систематической отладки в многокомпонентной системе — сперва добавить наблюдаемость на
  границе, прогнать раз, увидеть, ГДЕ рвётся. Конкретно:
  - имя логгера → `botkin.bot.upload`;
  - `poll_until_done`: маркеры `[POLL_START]/[POLL_STAGE]/[POLL_DONE]/[POLL_TIMEOUT]`;
  - `run_progress_flow` обёрнут в `try/except` с `log.exception` (`[FLOW_ERROR]` — ловит
    D1) + маркеры `[FLOW_FINAL]/[FLOW_CLAIM]/[FLOW_CARD]/[FLOW_LOST_RACE]` (D2);
  - `_edit`: `[EDIT_FAIL]` отделён от шумного «message is not modified».

  Маркеры спроектированы так, чтобы один прогон **развёл три гипотезы**: `[FLOW_LOST_RACE]`
  → D2, `[POLL_TIMEOUT]` → D3, `[FLOW_ERROR]` → D1.

- **Урок:** в системе из двух процессов на общей SQLite «`Doc processed`» в логе одного
  процесса не значит «доставлено». И самое коварное — **слепое пятно наблюдаемости**:
  неправильное имя логгера спрятало весь слой. Прежде чем чинить, убедись, что вообще
  ВИДИШЬ компонент. Темы для статьи: **«цена одной буквы в имени логгера»**; **«fan-in
  доставки: poll бота против push pipeline — кто claim'нул, тот и показал, а проигравший
  обязан не молчать (ветка без `else` = немое зависание)»**.

### Разрешение по логам: подтверждён D3 (+ деградация extract на бэкенде)

Один прогон с маркерами развёл гипотезы однозначно:

```
[POLL_STAGE] Doc 20 | стадия='normalizing' | 12.2s
[POLL_TIMEOUT] Doc 20 | таймаут 120.0s | опросов=60 | последняя стадия='normalizing'
[FLOW_FINAL] Doc 20 | final=None
```

Ни `[FLOW_ERROR]` (D1 исключён), ни `[FLOW_LOST_RACE]` (D2 исключён) — чистый **D3**:
бот сдаётся по таймауту 120 с, документ всё ещё в `normalizing`. Корень — на бэкенде:

```
[FAILED_EXTRACT] ...#стр2 | Elapsed: 207.48s | Error: Invalid JSON: EOF while parsing a list
```

Постраничный добор стр.2 шёл **207 с** и вернул оборванный JSON (модель генерила дубли до
упора `num_predict`, список не закрылся) → `ExtractionError` → статус `failed`. Бот к этому
моменту уже две минуты как отвалился. Урок: **таймаут UI должен превышать реальный потолок
обработки, а сама обработка — иметь свой потолок (timeout на VLM-вызов + защита от
генерации-мусора), иначе UI и бэкенд расходятся в разные стороны.**

### Сложность: TelegramBadRequest «can't parse entities» — `<` из нормы рушит /show

- **Симптом:** при `/show` и листании (`browse`) бот падает, результат не показывается:

  ```
  TelegramBadRequest: Bad Request: can't parse entities:
    Unsupported start tag "1.0)" at byte offset 2171
    File "show.py", line 32, in cmd_show
  ```

- **Корень:** `parse_mode=HTML` + референс **«< 1.0»** (базофилы). `_format_labs` экранирует
  имя/значение/единицу через `html.escape`, но **не `ref`** — `_format_ref` вставляет
  литералы `<`/`>` оператора нормы и свободный `ref_text`. Telegram читает `<1.0` как
  открывающий тег → отказ парсинга. Баг скрыт ровно на тех анализах, где норма
  односторонняя — а это частый случай (базофилы, СРБ, онкомаркеры).

- **Решение (TDD):** `_format_ref` оставлен текстовым helper'ом (его контракт — чистый
  текст, легко тестировать), экранирование сделано **на границе HTML** в `_format_labs`:

  ```python
  ref = _format_ref(r)
  ref = f" ({html.escape(ref)})" if ref else ""
  ```

  `html.escape("норма <1.0")` → `норма &lt;1.0` — Telegram показывает «< 1.0». Заодно
  закрылся и свободный `ref_text` от модели (defense-in-depth). Тест `test_show_labs.py`
  до фикса закреплял баг (`assert "<5.0" in out`) — исправлен на `&lt;5.0` + добавлен
  `test_ref_operator_html_escaped` (нет сырого `<1.0`, есть `&lt;1.0`). Остальные пути
  (`doc_title`, `clinic`, doctor-reports) уже экранированы — проверено.

- **Урок:** при ручной сборке HTML-сообщений экранировать надо **каждую** вставку
  переменных, а не только «очевидные». Один неэкранированный путь = краш на реальных
  данных. Тема для статьи: **«parse_mode=HTML и пользовательские данные: где `html.escape`,
  а где забыли — и почему тест, закрепляющий сырой `<`, маскировал баг»**.

### repro бага «< 1.0» (отдельно от доставки)

2. Вызвать `/show` или листать карточки (`browse`) → `TelegramBadRequest: can't parse entities`.
3. Фикс закрыт коммитом ниже; проверка — `pytest tests/test_show_labs.py`.

деградирует до 207 с и невалидного JSON). Попутно найден и **исправлен** независимый баг
HTML-экранирования нормы (`<` рушил `/show`). Тесты **183 passed**, ruff чисто.

## Итерация 4: лечим D3 послойно — extract не должен ронять документ, UI не должен сдаваться раньше бэкенда

Логи дали точный диагноз, и оказалось, что «зависание» — это сцепка четырёх дефектов в
двух процессах. Чинили каждый по TDD (RED → GREEN), от корня к UX.

### Сложность: одна деградировавшая страница убивала весь документ

- **Симптом:** общий вызов дал 6 строк (harvester), стр.1 — 1 строку, но стр.2 ушла в
  генерацию дублей на 207 с и вернула оборванный JSON → `ExtractionError`. Это исключение
  всплыло из цикла добора в `run_analysis` → `orchestrator` → статус `failed`. **Всё
  потеряно**, хотя 7 строк уже были извлечены.

- **Решение (фикс 1а, толерантность добора):** цикл постраничного добора оборачиваем в
  `try/except ExtractionError` — сбой страницы логируется (`[MULTIPAGE_PAGE_FAILED]`) и
  пропускается, уже собранные строки сохраняются. Тот же приём защищает и общий вызов:
  при его падении `rows=[], n_tables=0` → добор постранично ещё может спасти данные.

- **Урок:** в пайплайне «общий проход + добор частей» сбой ОДНОЙ части — ожидаемое
  событие, а не катастрофа. Изоляция сбоя на уровне части = частичный результат вместо
  полного провала.

### Сложность: модель генерит дубли до потолка и обрывает JSON

- **Симптом:** в сыром ответе MCV/MCH/MCHC/RDW повторяются по 2–3 раза; список раздувается
  до `num_predict` (8192), генерация не успевает закрыться → `EOF while parsing a list`.

- **Решение — три слоя защиты:**
  - **Фикс 3 (дедуп):** `_dedup_rows` схлопывает одинаковые `(имя, value_num, value_text)`
    на выходе `rows_from_raw` и `harvest_lab_rows` — повторы не доходят до БД и метрик.
  - **Фикс 2 (timeout VLM):** `get_raw_client` имел дефолт `timeout=600 с` — оттого вызов
    и висел 207 с. Ввели `VLM_REQUEST_TIMEOUT` (env, дефолт 120 с): деградировавший вызов
    прерывается, страница пропускается (фикс 1а), документ сохраняет остальное.
  - **Фикс 1б (salvage обрезанного JSON):** когда instructor падает на невалидном JSON,
    сырой текст последнего ответа достаётся из `InstructorRetryException.last_completion`
    и кладётся в `ExtractionError.raw_text`. `_salvage_json_objects` сканирует текст по
    балансу фигурных скобок (с учётом строк/escape) и вытаскивает все **полные**
    объекты-строки до точки обрыва → harvester собирает из них показатели
    (`[EXTRACT_SALVAGED]`). Полный массив невалиден, но первые N объектов — валидны.

- **Урок:** с локальной 8B-VLM «невалидный ответ» — норма, и относиться к нему надо как
  к потоку, который может оборваться в любой момент: парсить инкрементально, спасать
  префикс. Темы для статьи: **«три уровня обороны от генерации-мусора: дедуп → таймаут →
  salvage префикса»**; **«как достать сырой ответ из исключения instructor и зачем»**.

### Сложность: UI сдавался раньше бэкенда (корень D3)

- **Симптом:** `poll timeout` бота — хардкод 120 с, а обработка с постраничным добором
  легко переваливает за это. Бот возвращал `None` → «Обработка затянулась», пока бэкенд
  ещё работал (или падал) — два процесса жили по разным часам.

- **Решение (фикс 4):** `BOT_PROGRESS_TIMEOUT` (env), **увязан с потолком бэкенда** —
  дефолт `30 + 3 × VLM_REQUEST_TIMEOUT` (classify + общий + добор). Снизишь таймаут
  VLM — синхронно сожмётся и ожидание UI. `run_progress_flow` передаёт его в
  `poll_until_done`.

- **Урок:** таймауты связанных процессов нельзя задавать независимыми константами — у
  ведомого (UI) потолок должен **выводиться** из потолка ведущего (бэкенд), иначе они
  неизбежно разойдутся. Тема: **«согласованные таймауты: одна константа-источник вместо
  двух магических чисел»**.

### repro D3 после фиксов (ожидаемое поведение)

   с тем, что извлеклось (общий вызов + стр.1), без `failed`.
3. Бот ждёт до `BOT_PROGRESS_TIMEOUT` (≈390 с при дефолте) и показывает карточку.

Открыто для следующего живого прогона: подтвердить на реальной Ollama, что стр.2 теперь
либо извлекается, либо корректно деградирует без потери документа.

### Разрешение по живому прогону (Doc 21): доставка работает, вскрылось качество

Прогон подтвердил фиксы доставки фактом: `[POLL_START] timeout=390s` →
`[POLL_DONE] terminal='extracted' | 112.8s` → `[FLOW_CLAIM] claimed=True` →
`[FLOW_CARD] карточка показана`. Бот дождался (раньше сдавался на 120 с), карточка
отрисовалась, HTML-норма «<1.0» корректна. Salvage/timeout не активировались — стр.2
уложилась в 57.8 с с валидным JSON. **Баг A и D3 закрыты.**

Но карточка обнажила два бага уровнем глубже — недетерминизм модели и матчинг справочника:

- **Баг C — дубли с противоречивыми значениями между общим вызовом и добором.** Итог 38
  строк вместо ~21. Общий вызов harvester'ом собрал ВЕСЬ документ (21 строка: СРБ + ОАК),
  но добор постранично всё равно запустился (критерий «неполно»: `таблиц=1 < страниц=2`
  ложно сработал — harvester склеил обе страницы в одну таблицу). Хуже: на втором проходе
  модель выдала ДРУГИЕ числа (Гемоглобин «13.7 г/дл» в общем vs «143 г/л» на стр.2;
  Гематокрит 40.8 vs 42) — `_dedup_rows` по `(имя, value_num)` их не схлопнул, т.к.
  значения разные. Корень: критерий неполноты завязан на число таблиц, а не на покрытие;
  + недетерминизм значений модели между проходами.

- **Баг D — матчинг ФСЛИ подставляет неверный биоматериал/показатель (клинически опасно).**
  rapidfuzz берёт лучшее совпадение по названию, игнорируя биоматериал, а в ФСЛИ сотни
  вариантов одного теста: «Эритроциты» → «…в цереброспinальной жидкости»; «Гемоглобин» →
  «…в моче тест-полоской»; «C-реактивный белок» → «…в плевральной жидкости»; «Тромбоциты»
  → «**CD31+ клетки**». Для медданных это недопустимо: подмена крови на ликвор/мочу. Корень:
  нормализатор не фильтрует кандидатов по биоматериалу документа и дописывает метод/локус,
  которого в исходнике нет.

Темы для статьи: **«дедуп недостаточен, когда модель недетерминирована в ЗНАЧЕНИЯХ, а не
только в ключах — нужен критерий покрытия, а не повторный проход вслепую»**; **«fuzzy-матчинг
по медицинскому справочнику без фильтра по биоматериалу — генератор клинически опасных
подмен»**.

Итог: баги доставки закрыты, качество извлечения/нормализации — новый скоуп (баги C, D).

## Итерация 5: баг D — справочник правит опечатки, но не подмешивает биоматериал

Карточка показывала «Гемоглобин, массовая концентрация **в моче** с помощью тест-полоски»,
хотя в PDF — просто «Гемоглобин». Нормализатор клал в `canonical` поле `FULLNAME` из ФСЛИ
(с биоматериалом/методом), а fuzzy-матч выбирал произвольную specimen-запись из десятков.

### Сложность: fuzzy по ФСЛИ дописывает локус, которого нет в документе

- **Симптом:** «Эритроциты» → «…в цереброспинальной жидкости», «Тромбоциты» → «CD31+клетки»,
  + россыпь ложных ⚠️ (единица бралась из чужой specimen-записи: 1/мкл вместо ×10⁹/л).

- **Решение (по требованию: справочник — для опечаток и единиц, не для подмешивания):**
  - **Дедуп-таблица по `ANALYTE`.** Справочник свёрнут с 20720 specimen/метод-записей до
    **6211 показателей**: одна запись на показатель — чистое имя (`ANALYTE`, инвариант к
    биоматериалу), объединённые синонимы (`SHORTNAME`+`ENGLISHNAME`+`SYNONYMS`) и набор
    допустимых единиц (`UNIT`, расщеплённый по ';'/','). `FULLNAME`/`SPECIMEN` не храним.
  - **`canonical` = `ANALYTE`**, не `FULLNAME`. Подмешивание биоматериала исчезает в корне:
    «Гемоглобин»→«Гемоглобин», «Эритроциты»→«Эритроциты», «Гематкрит»→«Гематокрит» (опечатка
    исправлена), при неуверенном матче (MCV) — имя из документа.
  - **Единицы — из документа.** `unit_mismatch` сверяет распознанную единицу с НАБОРОМ
    допустимых (а не одной из случайной записи) → ложные ⚠️ уходят.
  - **Приоритет канонического имени над чужим синонимом.** «тромбоциты» — синоним записи
    «CD31+клетки», и при сборке ключей он «крал» показатель «Тромбоциты». Двухпроходная
    загрузка (сначала имена, потом синонимы на свободные ключи) → «Тромбоциты»→«Тромбоциты».
  - **Заголовок документа — по группе ФСЛИ** (биоматериал убрали): «Гематологические
    исследования» вместо «Анализ крови».

- **Урок:** медицинский справочник нельзя использовать как «обогатитель» — только как
  корректор. Разворачивать `FULLNAME` с биоматериалом = выдумывать данные, которых нет в
  исходнике. Правильный гранулярность — `ANALYTE` (показатель), а локус/метод/биоматериал
  принадлежат конкретному измерению, а не справочной норме. И: точное каноническое имя
  всегда должно бить чужой синоним. Темы для статьи: **«справочник-корректор vs
  справочник-обогатитель: почему второе для медданных опасно»**; **«дедуп 20k→6k: одна
  сущность-показатель вместо декартова произведения биоматериал×метод»**.

### Остаточные хвосты (точность матча, отложено)

- «Базофилы, %» → «Базофилы; Лейкоциты» — составное `ANALYTE`-имя в самой ФСЛИ (не подмена
  показателя, а кривое каноническое имя справочника).
- Полноценный фильтр кандидатов по контексту/специфике и баг C (дубли с расхождением значений
  между общим вызовом и постраничным добором) — отдельным шагом.

Итог итерации 5: подмешивание биоматериала устранено, опечатки названий правятся, ложные ⚠️
убраны, «Тромбоциты» больше не «CD31+клетки». Реестр пересобран (6211). **192 passed**, ruff чисто.

## Итерация 6: баг C — дубли с противоречивыми значениями между проходами

На Doc 21 карточка содержала **38 строк вместо ~21**: каждый показатель ОАК продублирован
с разными числами (Гемоглобин «13.7 г/дл» и «143 г/л»).

### Сложность: общий вызов + постраничный добор плодят конфликтующие дубли

- **Симптом:** общий вызов harvester'ом собрал весь документ (21 строка), но добор всё
  равно запустился (критерий «неполно» `таблиц=1 < страниц=2` ложно сработал — harvester
  склеивает страницы в одну таблицу). На втором проходе модель выдала ДРУГИЕ числа для тех
  же показателей. Слияние `_merge_dedup` по ключу `(имя, значение)` не схлопывало их —
  значения-то разные → 38 строк.

- **Решение:** слияние проходов **по имени показателя** (нормализованному), а не по
  `(имя, значение)`. Повторный показатель отбрасываем — доверяем первому (общему) проходу;
  добавляем только реально новые показатели (с потерянной страницы). Дубли с противоречием
  исчезают, добор по-прежнему спасает недостающее.

- **Урок:** при дедупликации недетерминированного источника ключ должен опираться на
  СТАБИЛЬНЫЙ инвариант (имя показателя), а не на то, что плавает между прогонами (значение).
  Иначе «защита от дублей» пропускает ровно те дубли, что опаснее всего — с расхождением.

Замечание о скорости: критерий запуска добора (`таблиц < страниц`) всё ещё срабатывает
ложно и тратит лишние VLM-вызовы, но благодаря дедупу-по-имени они безвредны для
корректности (дубли отбрасываются). Оптимизация критерия — в задаче про скорость.

Итог итерации 6: дубли с противоречивыми значениями устранены (38→~21). **194 passed**, ruff чисто.
