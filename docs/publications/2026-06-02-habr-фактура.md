# Dr. Botkin: локальный разбор медицинских документов фото→структура на одной видеокарте

> Рабочее название. Альтернативы: «Как я научил локальную vision-модель читать русские
> анализы и не сойти с ума от OCR», «Медкарта в SQLite: VLM-пайплайн без облака».

[КДПВ: телефон с фото анализа → стрелка → структурированная карточка в Telegram]

## 1. TL;DR

Telegram-бот принимает фото или PDF медицинского документа (лабораторный анализ, заключение
врача, справка), распознаёт его **локальной** vision-моделью `qwen3-vl:8b-instruct` через Ollama
и сохраняет **нормализованные** структурированные данные в SQLite. Названия лекарств сверяются
с офлайн-справочником ГРЛС на **20 948** названий (исправление ошибок распознавания,
заполнение МНН, статус регистрации). Никакие медицинские данные не покидают машину пользователя.

Стек: Python 3.12, FastAPI, aiogram 3, Ollama (qwen3-vl:8b-instruct), instructor + Pydantic,
OpenCV, pymupdf, RapidFuzz, SQLite (WAL). Целевое железо — ноутбук/ПК с NVIDIA GPU (отлажено на
**RTX 3080**, Ollama в WSL2). Разработка — TDD, **115 тестов**, ruff.

Под катом — полный разбор пайплайна `приём → препроцессинг → классификация → извлечение →
нормализация → хранение → доставка`, со всеми граблями: почему пришлось уйти от классического
OCR к vision-модели, почему `thinking`-вариант модели оказался ловушкой, как лечились
бесконечные циклы генерации и обрывы JSON, и почему на Windows бот сам ищет адрес WSL2.

## 2. Проблема и мотивация

Результаты анализов, выписки, заключения — это **персональные данные специальной категории**
(ст. 10 152-ФЗ). Их нельзя просто так слать в облачный OCR или чужой LLM-API. При этом дома
у человека копятся PDF из лабораторий и фото бланков из поликлиник, и хочется:

- складывать их в одно место и видеть **динамику** показателей (гемоглобин по годам и т.п.);
- не диагностировать, а **структурировать и хранить** — источник правды под рукой;
- делать это **без облака** — данные остаются на личной машине.

Отсюда главное архитектурное ограничение, заданное ещё до первой строчки кода: **весь разбор
— локальный**. Vision-модель крутится в Ollama на домашней видеокарте, справочник лекарств —
офлайн-файл в репозитории, база — один SQLite-файл. Внешняя сеть нужна только Telegram —
транспорту, через который пользователь общается с собственным ботом.

## 3. Принципы

- **Local-first / on-premise.** Ни один медицинский байт не уходит в сторонний сервис.
  Инференс — Ollama на локальном GPU; справочник и БД — на диске.
- **Structured output, а не свободный текст.** Модель обязана вернуть JSON по Pydantic-схеме
  (через `instructor`), иначе ответ отклоняется и перезапрашивается. Никакого парсинга прозы
  регекспами.
- **Восстановимость без потерь.** Рядом с нормализованными полями сохраняется `raw_extraction`
  — полный сырой ответ модели до нормализации. Если правила нормализации изменятся, данные
  можно пересобрать, ничего не потеряв.
- **Изоляция пользователей.** Все запросы к данным проходят через репозиторий, привязанный к
  `user_id` — один пользователь не видит чужие документы.
- **TDD.** Каждый блок — спека → план → реализация по красно-зелёному циклу. На момент фактуры
  — 115 тестов, ruff чист. Vision-модель в тестах мокается, поэтому тесты гоняются на CPU без
  GPU и без сети.

Два процесса:

1. **Бот** (`botkin.bot`, aiogram 3) — принимает файлы, шлёт их на backend, показывает живой
   прогресс и карточки документов, обслуживает навигацию (`/list`, `/period`, `/dynamics`,
   `/show`).
2. **Backend** (`botkin.api`, FastAPI) — принимает загрузку, валидирует, кладёт файл и запись в
   БД, запускает пайплайн обработки в `BackgroundTasks`.

Модули пакета `src/botkin/`:

| Модуль | Ответственность |
|---|---|
| `api/` | FastAPI-приложение, роут `/upload`, зависимости (user_id из заголовка) |
| `bot/` | aiogram-роутеры: `upload`, `browse`, `show`, `dynamics`, `period`, клавиатуры, карточки, прогресс |
| `pipeline/` | `orchestrator` (classify→extract→normalize→persist), `notifications` |
| `llm/` | `client` (Ollama через OpenAI-совместимый интерфейс), `classify`, `extract`, `prompts` |
| `preprocess/` | `images` (resize/CLAHE/unsharp/deskew), `formats` (определение типа по содержимому) |
| `normalize/` | `drugs` (ГРЛС-фаззи), `dates`, `units`, `numbers` |
| `db/` | `connection` (+миграции), `repos`, `queries`, `schema.sql` |
| `domain/` | Pydantic-модели (`LabResult`, `DoctorReport`, `ClassifyResult`) |
| `viz/` | `plots` (графики динамики через plotly+kaleido) |

## 5. Пайплайн по шагам

Точка входа — `POST /upload` (`api/routes/upload.py`). Дальше всё асинхронно в
`pipeline/orchestrator.py::process_document`.

### 5.1. Приём файла

- Принимаются `.pdf, .jpg, .jpeg, .png, .heic, .heif, .webp`; лимит **20 МБ**.
- Тип определяется **по содержимому**, а не по расширению (`preprocess/formats.py::resolve_extension`):
  iPhone присылает HEIC вообще без расширения в имени — имя файла лишь подсказка, решает
  сигнатура первых байтов.
- Имя обеззараживается (`/` и `\` → `_`), файл кладётся в `sources/<user_id>/<YYYY-MM>/<ts>-<имя>`.
- Создаётся запись в `documents` со статусом `received`, обработка ставится в `BackgroundTasks`.

### 5.2. Препроцессинг изображения (`preprocess/images.py`)

Размер картинки прямо определяет число vision-токенов и читаемость мелкого шрифта в бланках.
Поэтому два разных режима:

- **для классификации** — дёшево: длинная сторона ужимается до **1000 px**, только первая
  страница, без улучшений;
- **для извлечения** — качественно: длинная сторона приводится к **2200 px** (`upscale=True` —
  маленькие телефонные фото *растягиваются* до этого размера, большие ужимаются), затем
  CLAHE + unsharp + deskew.

Конкретика (из `config.json` / `config.py`):

| Параметр | Значение | Зачем |
|---|---|---|
| `extract_long_side` | 2200 | целевое разрешение для чтения |
| `classify_long_side` | 1000 | дешёвая классификация |
| `clahe_clip` | 2.0 (tile 8×8) | локальный контраст по L-каналу LAB |
| `unsharp_amount` | 1.5 | мягкая резкость (Gaussian blur + addWeighted) |
| `deskew_min_angle` | 3.0° | меньше — не выпрямляем |
| `deskew_min_area / max_area` | 0.40 / 0.97 | документ должен занимать 40–97% кадра |
| `pdf render_dpi` | 200 | рендер PDF-страниц через pymupdf |
| `max_pages` | 50 | предел страниц на документ |
| `jpeg_quality` | 90 | выход — JPEG в base64 |
| `lowres_warn` | 1500 | ниже — предупреждаем пользователя |

**Deskew** (`_doc_angle`) — best-effort: документ ищется как светлая (низконасыщенная) область
на более насыщенном фоне (HSV-saturation → Otsu-порог → морфология open 9×9, close 35×35 →
крупнейший контур → `minAreaRect`). Угол нормализуется в `[-45,45]°`; если контур не похож на
документ (доля площади вне 0.40–0.97) или угол меньше 3° — выпрямление **не** делается, берётся
полный кадр. Это сознательный консерватизм: лучше не трогать, чем повернуть бланк по бликам.
PDF не деперекашиваем — он уже ровный.

HEIC читается через `pillow-heif` (`register_heif_opener`), EXIF-ориентация снимается
`ImageOps.exif_transpose` (иначе телефонные фото приходят «лёжа»).

### 5.3. Классификация (`llm/classify.py`)

Дешёвый VLM-вызов по уменьшенной первой странице. Модель возвращает Pydantic-схему
`ClassifySchema`: `doc_type ∈ {analysis, doctor_report, certificate, unknown}`, `confidence`,
а также `title` (название документа как в источнике) и `clinic` (лаборатория/клиника) — это
сразу обогащает карточку.

### 5.4. Извлечение (`llm/extract.py`)

В зависимости от типа — `run_analysis` или `run_doctor_report`. Картинки готовятся в режиме
извлечения (2200 px, upscale+deskew+enhance), кладутся в сообщение как `image_url` с
base64-JPEG, модель принуждается вернуть список Pydantic-моделей:

- **анализ** → `list[LabResult]`: `analyte_name/code, value_num/value_text, unit,
  ref_low/ref_high, taken_at`, плюс «сырые» `value_raw/unit_raw/taken_at_raw`;
- **заключение** → `list[DoctorReport]`: `diagnosis, complaints[], anamnesis,
  recommendations[], medications[], visit_date, doctor_name, department`.

Промпты (`llm/prompts.py`) жёстко задают правила: извлекать только реальные показатели,
сохранять единицы и флаги как в документе (`«*», «↑», «↓», «(+)»`), диапазон в ячейке →
`ref_low/ref_high`, отсутствующее поле → `null`.

### 5.5. Нормализация и persist (`pipeline/orchestrator.py`)

- единицы → канон (`normalize/units.py::canonical_unit`, хранится и `unit_raw`);
- даты → ISO (русские даты, см. §6);
- названия лекарств в `medications` → сверка с ГРЛС (см. §6);
- сырой ответ модели целиком → `documents.raw_extraction`;
- статусы документа двигаются `received → recognizing → normalizing → extracted` (или `failed`).

### 5.6. Доставка результата

Гибрид: бот **поллит** статус и показывает прогресс-бар; backend после завершения ждёт
`DELIVERY_FALLBACK_DELAY = 130 с` (заведомо больше таймаута поллинга) и, если бот ещё не
«забрал» доставку, шлёт **push**. Кто доставил первым — определяется атомарным
`claim_delivery` (см. §10).

## 6. Нормализация и справочник ГРЛС

### 6.1. Лекарства (`normalize/drugs.py`)

Справочник `reference/drugs/registry.jsonl` — **20 948** названий (+1 строка `_meta`),
структурно: имя, тип (`trade`/`mnn`/`both`), связанное МНН, статусы регистрации, номера РУ.
Собран офлайн из официальной выгрузки ГРЛС (происхождение реестра — отдельная история раннего
под-проекта; для бота это готовый упакованный файл, рантайм полностью офлайн).

Матчинг через RapidFuzz:

- **scorer = абсолютная дистанция Дамерау-Левенштейна.** Выбран по замеру на словаре из 20 948
  имён: OCR-ошибки в названиях — это расстояние 1–3, и абсолютная дистанция ставит верный
  ответ первым, тогда как `WRatio`/`JaroWinkler` на большом словаре дают ложные снапы.
- **cap по длине:** `cap = max(1, floor(len(имя) × 0.40))` — короткие имена строже (меньше
  ложных совпадений), это `DRUG_MAX_EDIT_RATIO`.
- **ratio-floor:** дополнительно `fuzz.ratio ≥ 70` (`DRUG_RATIO_FLOOR`), иначе кандидат
  отклоняется.
- **правило безопасности:** не прошло порог — название **не подменяется**, сохраняется
  оригинал со статусом `unverified`. Редкие/неизвестные препараты не «исправляются» в
  похожие — это защита от ятрогенной ошибки.

Примеры коррекции (из README): `Элкап → Элькар`, `Глиалатин → Глиатилин`. На совпадении
заполняется МНН из торгового имени и подтягивается статус регистрации (действующий/исключён/
приостановлен) и номер РУ.

Для строк `medications` из заключения (свободный текст с дозой) — `correct_free_text`: отрезает
хвост с первой цифры/разделителя (`«- 500 мг»`), матчит ведущее имя, при неудаче — первое
слово; исходная строка всегда сохраняется как `raw`.

### 6.2. Даты, единицы, числа

- `normalize/dates.py` — русские даты в ISO; рядом хранится `taken_at_raw`.
- `normalize/units.py` — единицы к канону, рядом `unit_raw`.
- `normalize/numbers.py` — десятичная запятая → точка, с сохранением сырого значения.

Принцип один: **нормализуем, но сырое не выбрасываем**.

## 7. Модель и инференс

- **Модель:** `qwen3-vl:8b-instruct` в Ollama. Важно — именно **instruct**, не `thinking`
  (см. §10.1).
- **Клиент** (`llm/client.py`): Ollama через **OpenAI-совместимый** интерфейс (`OpenAI` SDK,
  `base_url=.../v1`, `api_key="ollama"`), поверх — `instructor` в `Mode.JSON` для принудительной
  валидации ответа по Pydantic-схеме. HTTP-таймаут — 600 с (vision-инференс на 8B небыстрый).
- **Опции Ollama** (`default_options`): `num_ctx=16384`, `num_predict=8192`,
  `repeat_penalty=1.2`, `keep_alive=30m` (модель не выгружается из VRAM между вызовами — не
  перезагружаем ~6 ГБ весов каждый раз). Temperature = 0.0.
- `instructor` вызывается с `max_retries=2` — если JSON не прошёл схему, запрос повторяется.
- **Железо:** отлажено на RTX 3080, Ollama в WSL2. Бот умеет автоопределять адрес WSL2 (§10.4).

## 8. Хранение

SQLite, прагмы из `schema.sql`:

```sql
PRAGMA journal_mode = WAL;     -- конкурентные чтения + одна запись
PRAGMA foreign_keys = ON;
PRAGMA busy_timeout = 5000;    -- ждать блокировку до 5 с, а не падать сразу
```

Таблицы: `users` (один Telegram-пользователь = один tenant), `documents` (с `raw_extraction`,
`title`, `clinic`, `delivered_at`, статусами-стадиями), `lab_results`, `doctor_reports`.
Индексы под реальные запросы: `documents(user_id, created_at)` — лента и навигация по соседям;
`lab_results(user_id, analyte_name, taken_at)` — динамика показателя.

Миграции (`db/connection.py`) — идемпотентные: `ALTER TABLE ... ADD COLUMN` по факту отсутствия
колонки, и аккуратное пересоздание `documents` при изменении `CHECK`-ограничения (с переносом
данных и переразметкой legacy-значений). Тип `prescription` (рецепты) был полностью удалён из
проекта — миграция дропает старую таблицу и переразмечает старые записи в `unknown`.

## 9. Telegram-UX

- **Приём:** фото и документы; для «фото» (Telegram жмёт до ~1280 px) бот советует прислать
  **файлом** ради полного разрешения, а при ширине ниже `lowres_warn=1500` усиливает
  предупреждение — и делает это **до** прогресс-сообщения, чтобы порядок сообщений был
  правильным.
- **Живой прогресс** (`bot/progress.py`): `poll_until_done` поллит статус и редактирует одно
  сообщение по смене стадии (`render_progress` — прогресс-бар стадий).
- **Карточки** (`bot/cards.py`): единая шапка (`title` + `clinic`) + детали; inline-навигация
  по соседним документам.
- **`/list`** — лента по 7 штук с фильтрами по типу (эмодзи-кнопки сеткой 2×2) и пагинацией.
- **`/period`** — пресеты и ручной ввод дат, документы и сводка показателей за период с трендом
  и сравнением с нормой.
- **`/dynamics <показатель>`** — график динамики (plotly + kaleido в PNG).

## 10. Инженерные грабли (подробно)

> Это — ядро статьи. Большинство решений рождалось из конкретных провалов.

### 10.1. `thinking`-вариант модели — ловушка скорости

Qwen3-VL существует в двух вариантах. **`thinking`-вариант перед ответом генерирует длинную
цепочку рассуждений** — для нашей задачи (прочитать бланк и вернуть JSON) это чистый оверхед:
время ответа растёт кратно, а качество структурированного извлечения не улучшается. Переход на
**`instruct`-вариант** дал кратное ускорение при том же результате. Урок зафиксирован прямо в
README как требование к установке: тянуть именно `qwen3-vl:8b-instruct`.
[ИЗМЕРИТЬ: thinking vs instruct — секунды на документ на RTX 3080]

### 10.2. Бесконечные циклы повторения во VLM

При нулевой температуре на сложных табличных бланках модель периодически **уходила в
бесконечную генерацию одной и той же фразы**, выедая контекст и зависая по таймауту. Лечение —
аппаратный штраф за повтор: `repeat_penalty = 1.2` во всех VLM-вызовах. Это убрало зависания,
сохранив детерминизм (temperature = 0.0).

### 10.3. Обрыв JSON на длинных бланках

Vision-модель на больших анализах генерирует длинный JSON, и при малом лимите токенов он
**обрывался на середине** — `instructor` не мог распарсить. Решение: явные щедрые лимиты
`num_predict = 8192` и `max_tokens = 8192`, плюс `max_retries=2` на стороне instructor. JSON
перестал отрезаться.

### 10.4. Windows → WSL2: бот не видит Ollama

Классическая ловушка loopback: бот на Windows, Ollama в WSL2. `127.0.0.1` из Windows не
пробрасывается в WSL2, а `localhost` в Python резолвится в IPv6 `::1` → `Connection Refused`.
Решение (`llm/client.py::_detect_ollama_url`): если заданный URL недоступен и мы на Windows —
динамически берём IP WSL2 (`wsl -d Ubuntu hostname -I`), формируем `http://<ip>:11434` и
проверяем доступность. Адрес кешируется.

### 10.5. Почему не классический OCR

Ранний путь был `OCR (распознавание текста) → текстовая LLM`. На табличных бланках **OCR
разваливал структуру** таблиц (терялись заголовки, склеивались значения), и текстовая модель
получала на вход «кашу», теряя показатели. На регрессионной выборке это давало точность ниже
порога. [ИЗМЕРИТЬ: точность OCR-пути vs VLM-пути на одной выборке — в ранней фактуре фигурировало
78.8% у OCR-пути; перепроверить применимость к текущему боту.] Вывод: для русских меддокументов
**vision-модель напрямую** (картинка → JSON), минуя OCR, оказалась устойчивее — она «видит»
таблицу как таблицу.

### 10.6. Препроцессинг под телефонное фото

Бланки фотографируют телефоном — под углом, при плохом свете, иногда в низком разрешении.
Отсюда связка upscale-до-2200 + CLAHE + unsharp + консервативный deskew (§5.2). Каждый шаг —
ответ на конкретную проблему читаемости, а deskew намеренно «трусливый» (строгие guard-условия
+ фолбэк на полный кадр), чтобы не испортить кадр ложным поворотом по бликам.

### 10.7. Навигация по соседям — индексом, а не загрузкой всего

Листание «предыдущий/следующий документ» сначала грузило все документы пользователя в память.
Заменено на индексный SQL-запрос соседа по `(user_id, created_at)` — O(log n) вместо загрузки
всей ленты (коммит `44647dc`).

### 10.8. Атомарная доставка финала

Прогресс показывается двумя путями (поллинг бота + push backend) — есть гонка «кто доставит
карточку». Решена атомарным `claim_delivery`: первый, кто проставил `delivered_at`, доставляет;
второй видит, что уже доставлено, и молчит. Никаких дублей карточки.

### 10.9. `instructor`: списки вместо wrapper-классов

Ранний опыт: громоздкие wrapper-схемы провоцировали у модели лишние рассуждения и
`IncompleteOutputException`. Переход на чистые списки Pydantic-моделей (`list[LabResult]`)
лёг на то, как Qwen возвращает массивы, и стабилизировал вывод.

## 11. Качество и процесс

- **TDD:** красно-зелёный цикл на каждый блок; vision-модель мокается → тесты идут на CPU без
  GPU/сети. На момент фактуры — **115 тестов** зелёные, ruff чист.
- **Процесс:** каждый блок — спека (`docs/superpowers/specs/`) → план (`docs/.../plans/`) →
  реализация по задачам. Блоки: A+B (распознавание + нормализация), A-препроцессинг,
  D (Telegram-UX), удаление рецептов, E (оптимизация навигации), F (эта публикация).
- **Достоверность фактуры:** ход разработки логировался как «фактура» для статьи; цифры в этом
  документе — из кода, неизмеренное помечено `[ИЗМЕРИТЬ]`.

## 13. Ограничения и планы

**Ограничения сейчас:**
- Рецепты (`prescription`) не поддерживаются — тип убран осознанно.
- Обработка сериализована глобальным семафором (`LLM_SEMAPHORE`) — одна VLM-задача за раз;
  при нескольких пользователях образуется очередь без видимого статуса «вы N-й».
- `BackgroundTasks` не переживают перезапуск backend — задача «в процессе» может зависнуть
  (нет реанимации залипших документов).
- Аутентификация backend — по заголовку `X-Telegram-User-Id` без подписи (безопасно только на
  localhost).

**Планы:**
- Реальные замеры (§12), графики.
- Реанимация залипших документов при старте, очередь со статусом.
- Возможный экспорт (CSV/PDF-отчёт), удаление документов пользователем.
