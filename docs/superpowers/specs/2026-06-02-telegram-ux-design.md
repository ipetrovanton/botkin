# Дизайн: блок D — Telegram UX

**Дата:** 2026-06-02
**Статус:** одобрено к реализации
**Зависит от:** блок A (распознавание), блок B (нормализация + ГРЛС) — оба в ветке `feature/pipeline-recognition-normalization`.

## 1. Цель и объём

Довести Telegram-UX до полноценного: живой прогресс обработки с авто-выдачей результата, навигация по документам inline-кнопками, запросы за период, корректный вывод данных ГРЛС (статус регистрации, уверенность сопоставления), а также обогащение документа метаданными (название и клиника).

**Входит:**
- Живой прогресс-бар обработки + авто-показ готового документа (поллинг статуса + push-fallback).
- Реальные стадии обработки в БД (`recognizing`, `normalizing`).
- Навигация: `/list` (список по 7), карточка документа, листание, фильтр по типу.
- Запросы за период: список документов и сводка показателей (пресеты + ручной ввод дат).
- Вывод ГРЛС в карточке назначений: МНН, предупреждение об отсутствии действующих регистраций, пометка неуверенного сопоставления.
- Обогащение метаданных: извлечение `title` (конкретное название документа) и `clinic` (медучреждение) через расширенный `classify`.

**Не входит:** полный редизайн БД (блок C), сквозной рефакторинг (E), документация для Хабра (F), действия «удалить/переобработать» над документом, кнопка повтора обработки.

## 2. Текущее состояние (отправная точка)

- Бот (`bot/main.py`) — aiogram polling. Команды: `/start`, `/help`, `/show` (последний документ), `/dynamics <показатель>`.
- Загрузка (`bot/handlers/upload.py`) шлёт файл на API, отвечает «Принято… подожди 30-60 с». Реального прогресса нет.
- API (`api/routes/upload.py`) создаёт документ, запускает фоновую `BackgroundTasks.add_task(process_document, …)`.
- Обработка (`pipeline/orchestrator.py`): статусы `received → processing → extracted/failed`. Промежуточных стадий нет. После завершения вызывает `notify_user` (push в Telegram) с текстом «обработано, нажмите /show».
- Бот **читает БД напрямую** через `db/queries.py` (`get_user_id`, `get_last_document`, `get_lab_results`, `get_prescriptions`, `get_doctor_reports`, `lab_dynamics`). Загрузку шлёт через HTTP API.
- `notify_user` (`pipeline/notifications.py`) создаёт `Bot` с `TG_BOT_TOKEN` и шлёт `send_message(chat_id=telegram_user_id, …)` — push-инфраструктура из backend уже существует.
- Данные ГРЛС **уже сохраняются** в `prescriptions`: `drug_mnn`, `drug_trade`, `match_status` (`matched`/`unverified`), `reg_statuses` (JSON-набор), `reg_numbers` (JSON).
- Метаданные `title`/`clinic` **не извлекаются и не хранятся** (в моделях лишь `doctor_name`, `department`).

## 3. Живой прогресс обработки

### 3.1. Стадии и статусы

Расширяем `documents.status` (миграция, расширение `CHECK`):

```
received → recognizing → normalizing → extracted
                                      ↘ failed
```

- `received` — документ создан в API (как сейчас).
- `recognizing` — идёт VLM (классификация + извлечение). Ставится в `orchestrator._run` перед `classify` (заменяет нынешний единый `processing`).
- `normalizing` — идёт нормализация и запись (`_persist_*`). Ставится после успешного `extract`, перед персистом.
- `extracted` — готово.
- `failed` — сбой на любой стадии.

### 3.2. Поллинг в боте

После загрузки `upload.py` запускает фоновую `asyncio.create_task`:

1. Шлёт первое сообщение, запоминает `message_id`.
2. Каждые **2 с** читает `documents.status` напрямую из SQLite (`get_document_status(doc_id, user_id)`).
3. Редактирует сообщение через `edit_message_text` **только при смене стадии** (а не каждый тик).
4. При `extracted` — заменяет на готовую карточку документа; ставит `delivered_at`.
5. При `failed` — заменяет на сообщение об ошибке.
6. Таймаут **120 с** — если статус не финальный, выводит «⏳ обработка затянулась, загляните позже через /show» и завершает задачу.

Рендер прогресс-бара — чистая функция `render_progress(status) -> str`:

```
⏳ Документ #9 — обрабатываю
📥 Принято ✓
🔍 Распознаю текст ●
🧩 Нормализую данные
✅ Готово
```

(пройденные стадии — `✓`, текущая — `●`, будущие — без маркера)

### 3.3. Push-fallback (гарантия доставки)

Чтобы результат дошёл, даже если бот перезапустился и поллинг-задача погибла:

- `documents += delivered_at TIMESTAMP` (NULL = ещё не показан пользователю).
- Поллинг-задача бота при показе финала выполняет атомарно: `UPDATE documents SET delivered_at = CURRENT_TIMESTAMP WHERE id = ? AND delivered_at IS NULL`.
- `orchestrator` после `set_status("extracted")` планирует отложенную проверку (внутри своей фоновой задачи): `await asyncio.sleep(POLL_TIMEOUT + grace)`, затем атомарно пытается захватить доставку тем же `UPDATE … WHERE delivered_at IS NULL`. Если захват удался (бот не показал) — шлёт результат через `notify_user`. Если строк не затронуто — бот уже доставил, push не нужен.
- Дубль практически исключён (атомарный захват), результат гарантированно доходит.

## 4. Навигация (inline-кнопки)

### 4.1. `/list` — список документов

- По **7** на страницу. Названия выводятся **в тексте сообщения** (не обрезаются), кнопки компактные.
- Фильтр по типу: `🧪 анализы / 💊 рецепты / 👨‍⚕️ заключения / Все`.
- Пагинация: `← Назад / Вперёд →`.

Макет:

```
📁 Твои документы (1–7 из 23)

1. 👨‍⚕️ Заключение невролога
   🏥 Клиника «Здоровье» · 02.06.2026
2. 🧪 Общий анализ мочи
   🏥 Инвитро · 28.05.2026
3. 💊 Рецепт
   🏥 — · 28.05.2026
…до 7

Фильтр:  [🧪] [💊] [👨‍⚕️] [Все]
[1] [2] [3] [4] [5] [6] [7]
[← Назад]            [Вперёд →]
```

- Название строки = `title` (распознанное конкретное название; fallback на лейбл `doc_type`, если `title` пуст).
- Кнопки `[1]…[7]` — выбор документа N на текущей странице → открывает карточку.
- Фильтр-кнопки покрывают `analysis`/`prescription`/`doctor_report`; документы типов `certificate`/`unknown` видны только в режиме «Все».

### 4.2. Карточка документа

Нажатие открывает полную карточку (переиспользуем общий рендер из `show.py`), с навигацией:

```
✅ Документ #22 · 🧪 Общий анализ мочи
🏥 Инвитро · 2026-05-28
────────────
• Глюкоза: 5.4 ммоль/л (норма 3.9–6.1)
• Холестерин: 6.8 ммоль/л (норма <5.2) ⬆️
…
────────────
[← Пред.]  [☰ К списку]  [След. →]
```

- Шапка показывает `title` и `🏥 clinic`.
- `← Пред. / След. →` — соседний документ по дате (в пределах текущего фильтра).
- `☰ К списку` — возврат к странице списка.

### 4.3. `/show`

Остаётся (последний документ), но под ним появляется тот же компонент клавиатуры навигации.

### 4.4. Кодирование callback

`callback_data` ≤ 64 байта:
- `doc:<id>` — открыть карточку.
- `lst:<type>:<offset>` — страница списка (`type` ∈ `a/p/d/all`).
- `nav:<id>:<dir>` — соседний документ (`dir` ∈ `prev/next`).
- `per:<preset>:<view>` — период (см. §5).

Кодирование/парсинг — в `bot/keyboards.py` (чистые функции, тестируемы).

## 5. Запросы за период

Команда `/period`:

1. Выбор диапазона быстрыми кнопками + ручной ввод:
   ```
   📅 За какой период?
   [Месяц] [3 месяца] [Год] [Всё время]
   ```
   Ручной ввод: `/period 2026-01 2026-03` (диапазон месяцев) или `/period 2026-01-01 2026-03-31`.
2. Выбор представления:
   ```
   📅 Май 2026 — что показать?
   [📁 Документы]   [📊 Показатели]
   ```

- **📁 Документы за период** — тот же список (по 7, пагинация, фильтр по типу), ограниченный диапазоном дат.
- **📊 Сводка показателей за период** — все анализы за диапазон, сгруппированы по показателю, тренд «первое → последнее», пометка нормы; кнопки → полный график (`/dynamics`):
  ```
  📊 Показатели за 3 месяца (по 12 анализам)
  ────────────
  Глюкоза: 5.4 → 5.1 → 4.9 ммоль/л  ✓ (норма 3.9–6.1)
  Холестерин: 6.8 → 6.2 ммоль/л  ⬆️ (>5.2)
  Гемоглобин: 145 → 148 г/л  ✓
  ────────────
  [📈 Глюкоза] [📈 Холестерин] [📈 Гемоглобин]
  ```

## 6. Вывод ГРЛС в карточке назначений

Данные уже в `prescriptions` — это правки **только вывода** (`_format_rx`).

Логика (на МНН-уровне, статусов несколько на один препарат):

- В наборе `reg_statuses` есть `active` → препарат действующий, **без пометки** (не шумим).
- Нет `active`, но есть `suspended`/`expired`/`excluded` → `⚠️ нет действующих регистраций в РФ`.
- `match_status == "unverified"` → `❓ распознано неточно (<ratio>%)`, показываем, с чем сопоставили.

Значения статусов реестра: `active`, `expired`, `excluded`, `suspended`, `modified`, `eaeu`, `foreign_pkg`, `pending`. Для предупреждения значимо отсутствие `active`.

Рег-номер ГРЛС **не выводим** (по решению пользователя).

Макет:

```
💊 Назначения · Документ #21
🏥 Клиника «Здоровье» · 28.05.2026
────────────
• Флуоксетин (Флуоксетин Ланнахер)
  20 мг, 1 капс утром, 365 дн.
• Левокарнитин (Элькар)  ❓ распознано неточно (78%) — проверьте
  300 мг/мл, 2.5–3 мл утром, 30 дн.
• Фенитоин  ⚠️ нет действующих регистраций в РФ
  …
```

## 7. Обогащение метаданных (title + clinic)

- `classify` VLM расширяется: возвращает `{doc_type, title, clinic}`.
  - `title` — конкретное название документа как в источнике («Общий анализ мочи», «МРТ головного мозга», «Пункция…»), свободный текст.
  - `clinic` — название медучреждения/лаборатории.
- `domain/models.py` — модель результата классификации получает поля `title`, `clinic` (Optional).
- `llm/prompts.py` — в `CLASSIFY`-промпт добавить инструкции по извлечению `title` и `clinic`.
- `documents += title TEXT, clinic TEXT` (миграция).
- `orchestrator` сохраняет `title`/`clinic` сразу после классификации (до извлечения деталей — попадут в карточку даже при `failed`-извлечении).
- Без дополнительного VLM-вызова — переиспользуем существующий проход классификации.

## 9. Обработка ошибок

- Поллинг-таймаут 120 с → сообщение «обработка затянулась», push-fallback досылает финал.
- `failed` → карточка с текстом ошибки (без кнопки повтора).
- Telegram `"message is not modified"` при `edit_message_text` → перехватываем, игнорируем.
- Callback на чужой/несуществующий документ → проверка `user_id`, ответ «документ не найден».
- Пустой список / период без данных → понятные заглушки.
- Устаревший callback после рестарта бота → graceful-ответ (по `callback_data` восстанавливаем состояние, не храним в памяти).
- Гонка `delivered_at` (поллинг бота vs orchestrator) → атомарный `UPDATE … WHERE delivered_at IS NULL`.

## 10. Тестирование (TDD, LLM мокается — dev-сервер без GPU)

- `db/queries.py`: новые запросы на временной БД (`list_documents` с фильтром/пагинацией, `count_documents`, `get_document` с проверкой `user_id`, `documents_in_period`, `labs_in_period` с группировкой).
- Чистые функции:
  - `render_progress(status)` для каждой стадии.
  - `_format_rx` с разными `reg_statuses` (есть/нет `active`, `suspended`) и `match_status` (`matched`/`unverified`).
  - кодирование/парсинг `callback_data` (round-trip, граница 64 байта).
  - парсинг периода (`/period 2026-01 2026-03`, пресеты, ошибочный ввод).
- Callback-хендлеры с моканым `Bot` и тестовой БД (открытие карточки, листание, фильтр, выбор периода).
- Расширенный `classify`: мок VLM возвращает `title`/`clinic` → проверка сохранения в `documents`.
- Поллинг-логика без реального `sleep` (инъекция «часов»/функции ожидания; прогон смены стадий по подменённому статусу).
- Push-fallback: атомарный захват `delivered_at` — бот захватил → push не шлётся; бот не захватил → шлётся.
- Миграция: применение к существующей БД добавляет колонки и статусы, старые данные целы.

## 11. Открытые допущения

- Поллинг читает ту же SQLite-БД, что пишет backend; на одном хосте (WSL2) — это безопасно (WAL). Для распределённого развёртывания позже понадобится API-эндпоинт статуса (вне scope D).
- `clinic`/`title` зависят от качества VLM; при отсутствии данных — graceful fallback (`title` → лейбл `doc_type`, `clinic` → «—»).
