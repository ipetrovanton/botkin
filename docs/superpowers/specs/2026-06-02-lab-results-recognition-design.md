# Распознавание лабораторных анализов: справочник ФСЛИ, нормализация и вывод в бот

**Дата:** 2026-06-02
**Ветка:** `feature/lab-results-recognition`
**Статус:** дизайн утверждён, готов к плану реализации

## Проблема

Лабораторные анализы **вообще не доходят** до пользователя в Telegram-боте. Диагностика по
реальной БД (`data/botkin.db`):

```
doctor_report / extracted : 2
unknown       / failed    : 2
lab_results строк         : 0   ← ни одного показателя не сохранено
```

Параллельно пользователь сообщает о потере данных и искажении названий: «часть информации
упускается, слова коверкаются, метрики распознаются неверно». Это три независимых дефекта,
наложенных друг на друга:

1. **Pipeline теряет анализы** — документы не классифицируются как `analysis` либо извлечение
   падает (нет ни одной строки в `lab_results`).
2. **Промпт извлечения неполон** — не описывает односторонние референсы (`<5.0`, `>120`),
   которые встречаются на большинстве бланков (см. `sample_020.pdf`: «С-реактивный белок 1.8 мг/л,
   реф. <5.0»). Такие нормы молча теряются.
3. **Рендер `show.py:_format_labs` дырявый** — показывает только `value_num` (текстовые
   результаты выводятся как `None`); печатает норму лишь когда заданы *обе* границы
   (`ref_low` и `ref_high`), поэтому односторонние нормы исчезают; игнорирует `ref_operator`
   и `ref_text`.

Названия же анализов не нормализуются вовсе — в отличие от препаратов (блок B), для которых
есть fuzzy-match по ГРЛС.

## Цель

Анализы надёжно проходят весь pipeline и выводятся в боте; распознанные названия
нормализуются по официальному справочнику ФСЛИ (каноническое имя + LOINC + код НМУ),
а единицы измерения сверяются со справочными.

## Источник справочника

Приложенный `Справочник лабораторных тестов.xlsx` — официальная выгрузка
**«Справочник лабораторных тестов» ФСЛИ** (OID `1.2.643.5.1.13.13.11.1080`, портал НСИ
Минздрава `nsi.rosminzdrav.ru`). Проверено через поиск: альтернативные источники
(apicrafter, НСИ REST API `api.n3health.ru/fnsi`, loinc.org) дают те же данные, но требуют
сети/токена. **Решение: строим оффлайн на xlsx**, как реестр ГРЛС. Обновление с НСИ
документируем как опциональную процедуру, в рантайм не тянем.

Структура листа «Справочник» (шапка — строка 2, 1-based; данные с строки 3): 18 колонок.
Значимые: `ID`, `LOINC`, `FULLNAME` (полное имя), `ENGLISHNAME`, `SHORTNAME` (краткое),
`SYNONYMS` (синонимы через `;`), `ANALYTE`, `UNIT` (единица), `SPECIMEN` (образец),
`METHODTYPE`, `SCALETYPE`, `TESTSTATUS` (статус), `GROUP` (группа тестов), `NMU` (код НМУ).

Объём: 20 727 тестов (20 586 актуальных, 93 устаревших, 48 новых), 15 групп,
7292 с синонимами, краткое имя у всех, LOINC у 6534. **В реестр включаем все 20 727**
(статус сохраняем в записи — старые бланки могут содержать устаревшие названия).

## Архитектура

Решение повторяет проверенную структуру блока B (нормализация препаратов): офлайн-скрипт
сборки `xlsx → registry.jsonl`, ленивый синглтон-нормализатор на `rapidfuzz`, интеграция в
`orchestrator._persist_lab`, новые колонки в `lab_results`. Каждый блок изолирован и
тестируется отдельно.

```
xlsx ──[scripts/build_analyte_reference.py]──> reference/analytes/registry.jsonl
                                                          │
                                            [normalize/analytes.py: AnalyteNormalizer]
                                                          │
pdf/img ─[classify]─> analysis ─[extract]─> LabResult ─[_persist_lab + normalize]─> lab_results (+LOINC/НМУ/unit-check)
                                                                                          │
                                                                              [show.py:_format_labs] ─> Telegram
```

### Блок A — Сборка реестра анализов

**Файл:** `scripts/build_analyte_reference.py` (по образцу `build_drug_reference.py`)
**Выход:** `src/botkin/reference/analytes/registry.jsonl`

Из xlsx (лист «Справочник») для каждой строки формируется запись:

```json
{
  "name": "Активированное частичное тромбопластиновое время",
  "short": "АЧТВ",
  "english": "APTT in Platelet poor plasma by Coagulation assay",
  "synonyms": ["АПТВ", "Activated partial thromboplastin time"],
  "loinc": "14979-9",
  "nmu": "A12.05.039",
  "unit": "с",
  "group": "Коагулогические исследования",
  "specimen": "Плазма крови бедная тромбоцитами",
  "status": "active"
}
```

- `TESTSTATUS` маппится: `Актуальный→active`, `Новый→new`, `Устаревший→deprecated`.
- `LOINC == "0"`/пусто → `loinc: null`.
- `SYNONYMS` разбиваются по `;`, чистятся.
- Ключ дедупликации — `normalize_key(FULLNAME)` (lower, ё→е, схлоп пробелов), как в ГРЛС.
- Поисковый индекс строится в рантайме (см. блок B), а не в файле — файл хранит чистые записи.
- Первая строка файла — `{"_meta": "ФСЛИ <имя файла> <кол-во>"}`.

Запуск: `uv run python -m scripts.build_analyte_reference --src "Справочник лабораторных тестов.xlsx" --out src/botkin/reference/analytes/registry.jsonl`

### Блок B — `AnalyteNormalizer`

**Файл:** `src/botkin/normalize/analytes.py` (по образцу `normalize/drugs.py`)

- `@dataclass AnalyteMatch`: `raw`, `canonical`, `loinc`, `nmu`, `group`, `expected_unit`,
  `status` (`"matched"`/`"unverified"`), `match_status` (статус теста: active/new/deprecated),
  `distance`, `ratio`.
- Класс `AnalyteNormalizer`:
  - При загрузке строит плоский индекс: каждая запись разворачивается в несколько поисковых
    ключей — `FULLNAME`, `SHORTNAME`, `ENGLISHNAME`, каждый синоним — все указывают на одну
    каноническую запись. Это покрывает короткие имена с бланков («АЧТВ», «СРБ», «WBC»).
  - `correct(raw_name) -> AnalyteMatch`: нормализация ключа (lower, ё→е), скоринг
    `rapidfuzz.distance.DamerauLevenshtein.distance` по всем поисковым ключам; cap расстояния
    `max(1, floor(len(query) * ANALYTE_MAX_EDIT_RATIO))`; фильтр `fuzz.ratio >= ANALYTE_RATIO_FLOOR`.
    Совпадение ниже порогов → `status="unverified"`, `canonical=raw` (как у препаратов — не
    «прилепляем» ложное совпадение).
  - Очень короткие ключи (≤3 символов: аббревиатуры) требуют **точного** совпадения
    нормализованного ключа — иначе fuzzy на 3 символах даёт мусор. Это правило отдельно
    протестировать.
- Пороги в `config.py`: `ANALYTE_MAX_EDIT_RATIO` (старт 0.35), `ANALYTE_RATIO_FLOOR` (старт 75).
  Финальные значения калибруются на реальных бланках в реализации.
- `load_default()` — ленивый синглтон, читает `registry.jsonl`.

### Блок C — Интеграция в pipeline + проверка единицы

**Файл:** `src/botkin/pipeline/orchestrator.py`

- Глобальный синглтон `get_analyte_normalizer()` (как `get_drug_normalizer`).
- В `_persist_lab` для каждого `LabResult`: `match = normalizer.correct(item.analyte_name)`.
- Проверка единицы: `unit_canon = canonical_unit(item.unit)`; если справочная `expected_unit`
  задана и не совпадает с распознанной (после канонизации обеих) → `unit_mismatch = True`.
- Сохраняем в новые колонки (см. блок D). `analyte_name` (как в документе) **не перезаписываем** —
  каноника идёт в отдельную колонку.

### Блок D — Схема БД

**Файл:** `src/botkin/db/connection.py` — декларативная миграция через `_MIGRATIONS`.

Добавить в `_MIGRATIONS["lab_results"]` колонки:

```python
"analyte_canonical": "TEXT",   # каноническое имя из ФСЛИ
"loinc": "TEXT",               # код LOINC
"nmu_code": "TEXT",            # код НМУ
"analyte_group": "TEXT",       # группа тестов
"match_status": "TEXT",        # matched | unverified
"unit_expected": "TEXT",       # ожидаемая единица из справочника
"unit_mismatch": "INTEGER",    # 0/1 — расхождение единицы
```

`schema.sql` (`CREATE TABLE lab_results`) дополняется теми же колонками для свежих БД.
`_persist_lab` и `INSERT` расширяются. `get_lab_results` в `queries.py` отдаёт новые поля.

### Блок E — Промпт извлечения

**Файл:** `src/botkin/llm/prompts.py` — дополнить `ANALYSIS_VLM_SYSTEM`:

- Правило односторонних референсов: `"<5.0"` → `ref_operator="<"`, `ref_high=5.0`, `ref_low=null`;
  `">120"` → `ref_operator=">"`, `ref_low=120`, `ref_high=null`.
- Текстовая норма («отрицательно», «не обнаружено») → `ref_text`.
- Явное «извлекай ВСЕ строки таблицы, не пропускай ни одного показателя».
- Уточнить, что числовое и текстовое значение взаимоисключающи (`value_num` XOR `value_text`).

**Риск `extra="forbid"`:** `LabResult` и `DoctorReport` имеют `model_config = ConfigDict(extra="forbid")`.
Если qwen3-vl вернёт лишнее поле — instructor уходит в retry и при неудаче бросает
`ExtractionError` → документ `failed`. В реализации: воспроизвести на `sample_020.pdf`, и если
подтвердится — сменить на `extra="ignore"` (лишние поля отбрасываются, извлечение не падает).

### Блок F — Починка рендера

**Файл:** `src/botkin/bot/handlers/show.py` — переписать `_format_labs`:

- Значение: `value_num` если есть, иначе `value_text`, иначе пропуск строки.
- Норма (хелпер `_format_ref`): двусторонняя `ref_low–ref_high`; односторонняя по
  `ref_operator` (`<5.0`, `>120`); текстовая `ref_text`.
- Маркеры отклонения ⬆️/⬇️ учитывают односторонние границы и оператор.
- Значок ⚠️ при `unit_mismatch`; каноническое имя/LOINC показываем при `match_status="matched"`,
  если отличается от распознанного (формат уточняется при реализации, минимально — не ломать
  существующий вид карточки).

То же касается группового рендера `cards.py:format_labs_summary` — проверить, что односторонние
нормы не ломают тренды (правка только при необходимости).

## Тестирование (TDD)

Каждый блок — тесты до кода, в стиле существующих (`tests/test_normalize_drugs.py`,
`tests/test_llm_calls.py`, `tests/test_migration.py`):

- **A** `test_build_analyte_reference.py`: парсинг строки xlsx → запись; маппинг статусов;
  разбор синонимов; `LOINC=="0"`→null; реальная сборка (несколько строк фикстуры).
- **B** `test_normalize_analytes.py`: точный матч (distance=0); OCR-опечатка (distance 1–2);
  матч по синониму и краткому имени; аббревиатура требует точного совпадения;
  неизвестный анализ → `unverified` (не прилеплен); ratio-floor отбраковывает; `raw` сохраняется.
- **C** `test_orchestrator.py` (расширить): `_persist_lab` пишет каноника/LOINC/НМУ;
  `unit_mismatch` выставляется при расхождении единицы; `analyte_name` не перезаписан.
- **D** `test_migration.py` (расширить): новые колонки добавляются к старой `lab_results`;
  идемпотентность повторной миграции.
- **E** `test_prompts.py` (расширить): `ANALYSIS_VLM_SYSTEM` содержит правила односторонних
  референсов и `value_text`; модели парсят ответ с лишним полем (после смены на `extra="ignore"`).
- **F** `test_show_labs.py` (новый): текстовый результат рендерится (не `None`);
  односторонняя норма `<5.0` отображается; маркер по односторонней границе; `unit_mismatch`→⚠️.

Целевое состояние: все существующие тесты (115+) зелёные, ruff чист, плюс новые.

**Верификация на живой VLM.** Ollama в среде разработки недоступен, поэтому весь код
покрывается детерминированными mock-тестами (как `test_llm_calls.py`). Реальный прогон
`sample_020.pdf` (и других бланков) через живую Ollama выполняет пользователь на своей машине
и присылает результат/логи — по ним калибруются пороги нормализатора (блок B), правила
промпта (блок E) и подтверждается корень дефекта pipeline (блок «Диагностика»).

## Вне объёма (YAGNI)

- Подкачка справочников с `fedlab.ru`/НСИ в рантайме (только документируем процедуру обновления).
- Референсные нормы по полу/возрасту (xlsx ФСЛИ их не содержит; нормы берём из самого бланка).
- Группировка показателей по разделам в карточке (поле `analyte_group` сохраняем — UI позже).
- Распознавание `prescription` (снято с поддержки ранее).
