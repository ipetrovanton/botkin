# Дизайн: блок A+B — Pipeline распознавания + Нормализация

- **Дата:** 2026-06-02
- **Статус:** на согласовании
- **Область:** первый цикл декомпозиции (см. ниже). Закрывает пункты ТЗ 1, 2, 3, 4, 6 (частично).
- **Не входит:** редизайн БД целиком (блок C), Telegram UX (D), сквозной рефакторинг (E), документация для Хабра/конференции (F).

---

## 1. Контекст

`botkin` — Telegram-бот + FastAPI-бэкенд: принимает фото/PDF медицинских документов,
распознаёт их vision-моделью **qwen3-vl** (Ollama в WSL2 на ThinkPad RTX 3080 16 ГБ),
сохраняет структурированные данные в SQLite.

Текущий поток (`pipeline/orchestrator.py`): `classify` (1 VLM-вызов по 1-й странице) →
`extract` (2-й VLM-вызов, грузит **все страницы** в масштабе 2x одним запросом) → `persist`.

### Подтверждённые проблемы

1. **Латентность >60 с.** Два последовательных VLM-вызова; `extract` шлёт все страницы в
   2x-масштабе при `num_ctx=32768`; промпты борются с «размышлениями» модели — косвенный
   признак, что используется thinking-вариант (катастрофически медленный).
2. **Ошибки в названиях лекарств** (на живых `.txt`-выхлопах): `Элкап`→**Элькар**,
   `Глиалатин`→**Глиатилин**, `Флюоксетин`→**Флуоксетин**, `Тритико`→**Триттико**.
   Причина: **нет этапа нормализации/сверки со справочником**.
3. **PDF** обрабатывается «в лоб», без постраничной подготовки и контроля разрешения.
4. **Нормализация дат/единиц/чисел** примитивна или отсутствует; нет гарантии сохранности
   полей оригинала.

### Решения, согласованные на брейншторме

- Объём: **поэтапно**, спек на блок. Первый цикл — A+B.
- Модель: **`qwen3-vl:8b-instruct`**, цель латентности **<25–30 с**.
- Точность: **офлайн-справочник МНН + все доступные торговые** наименования (ГРЛС/ЕАЭС) +
  фаззи-коррекция; **редкие/неизвестные препараты не подменяются** — оригинал сохраняется,
  поле помечается «не сверено».
- Структура вызовов: **A3** — дешёвый `classify` + точный `extract`.

### Сверка с источниками (rule 04)

- qwen3-vl `instruct` vs `thinking`: thinking-вариант радикально медленнее
  (8B Thinking ≈ 262 с на задаче) — [Medium, 17.10.2025].
- `qwen3-vl:8b-instruct`: Q4_K_M, 6.1 ГБ, дефолт `temperature 1, top_k 20, top_p 0.95`
  — [ollama.com/library/qwen3-vl:8b-instruct].
- Баг производительности qwen3-vl на Windows относительно qwen2.5-vl —
  [github.com/ollama/ollama/issues/12854].
- Справочник (собрано 2026-06-02): официальная ZIP-выгрузка ГРЛС (`grls2026-06-02-1.zip`,
  8 листов-статусов), парсинг openpyxl → `registry.jsonl`, **20 948** уникальных названий
  (≈18 396 торговых, ≈3 689 МНН) со статусами, МНН-связкой и рег-номерами.
- Фаззи-матчинг: **RapidFuzz 3.14.5**, MIT, C++-бэкенд, Python ≥3.10
  — [pypi.org/project/RapidFuzz]. Scorer: дистанция Дамерау-Левенштейна (выбран по замеру).

## 2. Цели и критерии успеха

| # | Цель | Критерий приёмки |
|---|------|------------------|
| G1 | Снизить латентность | Типовой 1–2-страничный документ обрабатывается **<30 с** end-to-end на целевой машине |
| G2 | Исправлять названия лекарств | Золотые кейсы (Элькар/Глиатилин/Флуоксетин/Триттико) нормализуются корректно в тестах |
| G3 | Не терять данные оригинала | Для каждого документа сохранён сырой JSON извлечения; для нормализованных полей сохранено сырое значение |
| G4 | Не портить редкие препараты | При отсутствии уверенного совпадения название НЕ подменяется, помечается `unverified` |
| G5 | Единый формат дат/единиц | Даты → ISO; единицы → канон; десятичная запятая → точка; сырое значение сохранено |
| G6 | Чистая подготовка изображений | PDF и фото приводятся к контролируемому разрешению/качеству перед VLM |

## 3. Архитектура блока A (скорость)

### 3.1. Переход на instruct-вариант

- `config`: модель по умолчанию → `qwen3-vl:8b-instruct` (переопределяемо через env/config.json).
- Из промптов (`llm/prompts.py`) убрать анти-thinking костыли («Запрещено писать
  thinking/reasoning…», требования к Markdown-блокам ```json) — instruct-вариант не уходит
  в рассуждения, а structured output обеспечивает `instructor`.

> ### Почему так
> - **Альтернативы:** (A) оставить дефолтный тег `qwen3-vl:8b` — но он неоднозначен по
>   варианту и промпты приходится «лечить»; (B) сменить модель на qwen2.5-vl — но пользователь
>   выбрал держать 8b.
> - **Выбрано:** явный пин `instruct` — единственный надёжный способ исключить thinking-режим,
>   а это главный множитель латентности.
> - **Компромисс:** теряем reasoning-способности (нам они в извлечении не нужны).
> - **Когда пересмотреть:** если на целевой машине instruct не уложится в <30 с — рассмотреть
>   4b-instruct для простых типов (это уже двухрежимность, отдельный блок).

### 3.2. Препроцессинг изображений — новый модуль `preprocess/`

`preprocess/images.py`: единая функция подготовки страниц документа к VLM.

- **PDF** (`pymupdf`): рендер каждой страницы в растр с целевым **DPI** (вместо «scale 2x
  вслепую»), затем приведение длинной стороны к диапазону `IMAGE_MAX_LONG_SIDE`
  (ориентир 1600–2000 px) — иначе телефонные фото/сканы раздувают число vision-токенов.
- **Изображения** (jpg/png/heic/webp): декод (Pillow; HEIC — через доступный декодер),
  даунскейл по той же логике, нормализация ориентации по EXIF.
- Выход — список JPEG-байтов с управляемым `IMAGE_JPEG_QUALITY`.
- Все пороги (`IMAGE_MAX_LONG_SIDE`, `IMAGE_JPEG_QUALITY`, `PDF_RENDER_DPI`, `MAX_PAGES`) —
  в `config`, без магии в коде.

> ### Почему так
> - **Альтернативы:** слать оригиналы как есть — но размер картинки прямо определяет
>   число vision-токенов и латентность; чем крупнее, тем дольше и дороже по VRAM.
> - **Выбрано:** контролируемый даунскейл к «достаточному для OCR» разрешению.
> - **Компромисс:** на очень мелком шрифте агрессивный даунскейл может срезать детали —
>   поэтому порог консервативный (1600–2000 px), подбирается замером на золотых примерах.
> - **Когда пересмотреть:** если точность на мелком тексте падает — поднять порог или
>   добавить tiling длинных страниц.

### 3.3. Структура вызовов (A3)

- `classify`: уменьшенная 1-я страница (отдельный малый порог разрешения), крошечный бюджет
  вывода (`doc_type` + `confidence`), малый `num_ctx`. Цель — 1–3 с.
- `extract`: полноразмерные (после препроцессинга) страницы, схема под тип документа.
- Изображения для `extract` кодируются один раз.

### 3.4. Параметры Ollama

- `keep_alive` держит модель загруженной между вызовами и документами (убирает перезагрузку
  весов 6 ГБ) — выносим в `config`.
- `num_ctx` снизить с 32768 до реально необходимого (картинки + промпт + вывод); стартовое
  значение — в `config`, финал подбирается замером.
- `repeat_penalty` брать из `VLM_REPEAT_PENALTY` (сейчас захардкожен 1.2 в `classify`/`extract` —
  багфикс), `num_predict` пробросить из config (сейчас в config.json есть, но не читается).
- Семафор `LLM_SEMAPHORE = 1` сохраняем (одна GPU, параллельные VLM-вызовы только вредят).

## 4. Архитектура блока B (нормализация)

### 4.1. Новый слой `normalize/`

Этап **между `extract` и `persist`** в orchestrator. Подмодули с одной ответственностью:

- `normalize/dates.py` — парсинг множества форматов (`dd.mm.yyyy`, `dd/mm/yy`, `dd.mm.yy`,
  «23 марта 2026 г.», ISO, `yyyy-mm-dd`) → единый ISO-`datetime`. Сырое значение возвращается
  отдельно. Заменяет наивный `parse_ru_date` в `domain/models.py`.
- `normalize/units.py` — канонизация единиц (`×10⁹/л` / `10^9/L` / `тыс/мкл` → канон;
  таблица соответствий в `reference/`).
- `normalize/numbers.py` — десятичная запятая → точка, чистка пробелов/символов.
- `normalize/drugs.py` — фаззи-сверка названий со справочником (ядро блока B).

### 4.2. Справочник лекарств и фаззи-коррекция

- **Источник** — официальная **ZIP-выгрузка ГРЛС** (`grls2026-06-02-1.zip`, 8 XLSX-листов по
  статусам: Действующий, Изменённый, Исключённый, Истёкший, Выдано по правилам ЕАЭС, Действует
  (на подтверждении), Приостановлено применение, Действует (в иностранных упаковках)). Самый
  полный и авторитетный список; предоставляется пользователем. (Зеркало hubofdata/Wikidata из
  предыдущей итерации отброшены как менее полные.)
- **Структурированный справочник** `reference/drugs/registry.jsonl` (build-скрипт
  `scripts/build_drug_reference.py`, openpyxl): по одной записи на уникальное нормализованное
  имя (торговое и/или МНН). Поля записи:
  - `name` — каноничное написание (как в реестре),
  - `type` — `trade` | `mnn` | `both`,
  - `mnn` — связанное МНН (для торговых; позволяет заполнить МНН из распознанного торгового),
  - `statuses` — список статусов-списков, где встречается имя (`active`, `excluded`,
    `suspended`, `expired`, `eaeu`, `modified`, `pending`, `foreign_pkg`),
  - `reg_numbers` — номера регистрационных удостоверений (для торговых).
  Собрано на момент написания: **20 948** уникальных названий (≈18 396 торговых, ≈3 689 МНН),
  ~3.9 МБ. Коммитится в репозиторий (офлайн-работа рантайма). XLSX-колонки: рег-номер (2),
  торговое (8), МНН (9, `~`=нет); статус берётся из имени листа.
- **Назначение статусов/рег-номеров** (требование на будущее): даёт возможность подсветить, что
  назначенный препарат находится в списке «исключён»/«приостановлено применение»/«истёк», и
  хранить номер регистрационного удостоверения.
- **Алгоритм коррекции** (`normalize/drugs.py`, RapidFuzz). Выбор scorer — по замеру на
  собранном словаре (11k названий), см. блок «Почему так»:
  1. Нормализуем распознанное имя (lower, ё→е, trim).
  2. Лучшее совпадение по **абсолютной дистанции Дамерау-Левенштейна** с лимитом, зависящим
     от длины: `cap = max(1, floor(len(q) * DRUG_MAX_EDIT_RATIO))`.
  3. Дополнительный фильтр по `fuzz.ratio(q, candidate) ≥ DRUG_RATIO_FLOOR` (отсекает
     совпадения «в пределах cap», но непохожие по форме).
  4. **Если оба условия выполнены** → `matched`: возвращаем запись справочника (`canonical`,
     `type`, `mnn`, `statuses`, `reg_numbers`); сырое сохраняем в `drug_raw`. Для торгового
     названия дополнительно **заполняем МНН** из связанного `mnn` записи.
  5. **Иначе** → **НЕ подменяем**: `drug_raw` = оригинал, статус `unverified` (защита редких
     препаратов — G4).
- Константы `DRUG_MAX_EDIT_RATIO` (старт 0.40) и `DRUG_RATIO_FLOOR` (старт 70) — в `config`.
- **Чистые поля рецепта** (`prescription.drug_mnn`/`drug_trade`) сверяются напрямую. Для
  **свободного текста** `doctor_report.medications` (строки с дозой/формой, напр.
  «Элкап - 300 мг/мл…») — best-effort: `correct_free_text(line)` отрезает хвост с первой цифры/
  разделителя, берёт ведущее имя и сверяет его. Recall ниже (ведущие глаголы вроде «Уменьшить
  Триттико» могут не распознаться), но **оригинал сохраняется всегда**, при неуверенности —
  `unverified`. Улучшение разбора свободного текста — отдельная итерация.

> #### Замер scorer'ов (2026-06-02, словарь 20 948 названий)
> На полном словаре `WRatio` ранжирует мусор выше верного (`элкап`→«элапраза» 80), а
> `JaroWinkler` не отделяет мусор (85) от `элкап`→`элькар` (86). **Дистанция
> Дамерау-Левенштейна** ставит верный ответ первым для всех золотых кейсов: реальные
> OCR-ошибки = расстояние 1–3 (`элкап`→`элькар` 2, `Глиалатин`→`глиатилин` 3,
> `Флюоксетин`→`флуоксетин` 1, `тритико`→`триттико` 1), мусор = 13. Правило
> «Дамерау-cap + ratio-floor» дало **13/13** на проверочном наборе (включая отказ для
> несуществующего препарата).

> ### Почему так
> - **Альтернативы:** (A) онлайн-сверка ГРЛС — отвергнута (сеть, зависимость от сервиса);
>   (B) только МНН — не исправит торговые названия, а реальные ошибки именно в них;
>   (C) только промпт — не убирает ошибки распознавания мелкого/рукописного текста.
> - **Выбрано:** офлайн-справочник + RapidFuzz с дистанцией Дамерау-Левенштейна
>   (cap по длине) и ratio-floor, с правилом «не уверен — не трогай». Scorer выбран по замеру
>   (см. блок «Замер scorer'ов»): `WRatio`/`JaroWinkler` на большом словаре дают ложные
>   совпадения, Дамерау — нет.
> - **Компромисс:** короткие названия (≤4 симв.) с cap=1 ловят меньше OCR-ошибок; зато нет
>   ложных «снапов». `drug_raw` хранит оригинал в любом случае.
> - **Когда пересмотреть:** если на реальных данных вырастут пропуски на длинных названиях —
>   тюнинговать `DRUG_MAX_EDIT_RATIO`/`DRUG_RATIO_FLOOR` или ввести сверку с учётом контекста
>   МНН (двухступенчато: торговое → его МНН).

### 4.3. Гарантия сохранности данных (G3)

- В `documents` добавляется `raw_extraction` (TEXT, JSON) — **полный сырой ответ модели** до
  нормализации. Любое поле всегда восстановимо.
- Для нормализуемых полей храним пару «сырое + нормализованное» (см. §5).

## 5. Минимальные изменения схемы БД

Полный редизайн — блок C. Здесь только необходимое для A+B, чистой миграцией
(идемпотентный `ALTER TABLE ... ADD COLUMN`, безопасно для существующей БД):

- `documents`: `+ raw_extraction TEXT` (сырой JSON извлечения).
- `prescriptions`: `+ drug_raw TEXT` (что прочла модель), `+ match_status TEXT`
  (`matched` / `unverified`), `+ reg_statuses TEXT` (JSON-массив статусов из справочника),
  `+ reg_numbers TEXT` (JSON-массив номеров РУ). `drug_mnn` дозаполняется из связанного МНН
  справочника, если в документе было только торговое название.
- `doctor_reports`: `+ medications_normalized_json TEXT` (best-effort нормализация строк
  `medications`: для каждой — сырое + canonical/статусы/`unverified`).
- `lab_results`: `+ value_raw TEXT`, `+ unit_raw TEXT`, `+ taken_at_raw TEXT` (сырые значения).
- Доменные Pydantic-модели расширяются соответствующими полями.

> ### Почему так
> - **Альтернативы:** дождаться редизайна в блоке C — но тогда A+B не выполнит G3/G5
>   (сохранность оригинала), а это половина ценности блока.
> - **Выбрано:** аддитивные колонки сейчас, полноценная нормализация схемы — в C.
> - **Компромисс:** временно немного «широкие» таблицы; в C приведём к финальному виду.
> - **Когда пересмотреть:** при старте блока C.

## 7. Тестирование (без сети, без GPU)

> **Ограничение dev-окружения.** Машина разработки **не имеет GPU и моделей Ollama**.
> VLM-код здесь **не запускается**, латентность (G1) меряет пользователь локально.
> Поэтому **все автотесты мокают LLM-клиент** (`botkin.llm.client`); проверяемое в CI —
> препроцессинг, нормализация, парсинг дат, фаззи-коррекция, миграция, импорты. Запуск
> модели и бенчмарк скорости — вне CI, на целевой машине.


- `normalize/dates`: таблица форматов → ожидаемый ISO; мусор → `None` + сырое сохранено.
- `normalize/units`/`numbers`: канонизация вариантов.
- `normalize/drugs`: **золотые кейсы** из реальных `.txt` (`Элкап`→`Элькар`, `Глиалатин`→
  `Глиатилин`, `Флюоксетин`→`Флуоксетин`, `Триттико`); + кейс редкого препарата (нет в
  справочнике → `unverified`, оригинал не тронут); + кейс защиты от ложного «снапа».
- `preprocess/images`: PDF→N изображений, контроль длинной стороны и формата (на маленьком
  фикстур-PDF/PNG).
- Справочник в тестах — маленький фикстурный лукап (не тянем полный ГРЛС в CI).

## 8. Риски и открытые вопросы

- **R1.** Справочник собирается из выгрузки ГРЛС, которую периодически обновляет пользователь
  (формат листов/колонок может измениться). Митигировано: build-скрипт изолирует разбор XLSX,
  результат (`registry.jsonl`) коммитится, рантайм офлайн. Сам ZIP (~18 МБ) в репозиторий не
  кладём — это вход build-скрипта.
- **R2.** Подбор `num_ctx`/`IMAGE_MAX_LONG_SIDE`/`DRUG_FUZZY_THRESHOLD` требует замеров на
  целевой машине — это итеративная настройка после первичной реализации.
- **R3.** Точная цель <30 с зависит от железа и бага qwen3-vl на Windows (R3 в источниках);
  если не уложимся — фолбэк-варианты в §3.1 («Когда пересмотреть»).
- **O1.** HEIC-декодирование на Windows — выбрать конкретный декодер (`pillow-heif`?),
  уточнить при реализации (rule 04).
