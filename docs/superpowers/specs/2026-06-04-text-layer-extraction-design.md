# Извлечение лабораторных показателей из текстового слоя PDF

**Дата:** 2026-06-04
**Ветка:** feature/lab-results-recognition
**Статус:** дизайн утверждён, готов к плану реализации
**Scope:** только `doc_type == "analysis"`

## Проблема

Пайплайн извлечения построен на VLM-over-image: PDF рендерится в сжатый JPEG и
отдаётся модели `qwen3-vl:8b-instruct`. На цифровых PDF (ИНВИТРО и т.п.) это даёт три
класса дефектов, подтверждённых на живом прогоне `sample_020.pdf` (Doc 22/23/25):

1. **Галлюцинация-нормализация значений.** При жадном декодировании (`temp=0`) модель
   идёт по самому вероятному токену, а для известных показателей это заученная
   «каноническая» норма, а не увиденное. Пример: Гемоглобин в документе
   `13.7 г/дл, реф 11.7–15.5` модель вернула как `137 г/л, реф 120–160`. Сырой ответ
   VLM в логе уже содержит подменённые значения — то есть подмена рождается **внутри
   VLM-вызова**, а не в постобработке (harvester/маппер/нормализатор).
2. **Неполнота.** Модель обрывает длинную таблицу примерно на 13-й строке. Из 21
   показателя (СРБ + 20 ОАК) извлекается 14; теряются Базофилы %, весь блок «абс.»
   (5 показателей) и СОЭ.
3. **Обрыв значения.** Последняя извлечённая строка обрезается:
   Эозинофилы получили `value_raw = "3. "` вместо `3.1`, без единицы и референса.

Подбор параметров (`temp=0.3`, `repeat_penalty=1.2`, `num_predict=16384`) на одном
прогоне убрал галлюцинацию значения (Гемоглобин стал верным), но:
- это недетерминированно — `temp=0.3` означает «обычно верно», не «верно»;
- не лечит неполноту (по-прежнему 14/21) и обрыв.

### Корень

Цифровой PDF несёт **точный текстовый слой** (значения дословно, с десятичными,
правильными единицами `г/дл`, правильными референсами). Пайплайн его не касается:
`get_text()` нигде нет; `pymupdf` используется только для рендера в картинки
(`preprocess/images.py`). Мы выбрасываем готовый эталон и заменяем его галлюцинацией.

## Решение

Для PDF с годным текстовым слоем извлекать показатели **из текста**, а не из картинки.
Источник цифр меняется с «модель угадала по пикселям» на «модель переписала из точного
текста». Это:
- убирает галлюцинацию значений (цифры лежат в контексте модели как токены);
- даёт полноту (текст содержит все строки);
- детерминированно (текст читается одинаково каждый раз);
- открывает **verbatim-страж** — проверку, что каждое число модели есть в исходном
  тексте (невозможна при картинке на входе).

Структурирование текста в строки делает модель, а не шаблон: разные лаборатории имеют
разные раскладки колонок, шаблонный парсер не генерализуется. Геометрический детектор
таблиц `pymupdf.find_tables()` на ИНВИТРО (нет линий таблицы) выделил лишь 3 строки —
тоже непригоден. Поэтому: детерминированно собираем физические строки по координатам,
а разметку колонок отдаём модели.

## Архитектура

Точка входа неизменна: `extract.run_analysis(path) -> list[LabResult]`. Текстовый слой
меняет только источник строк внутри неё. Три изолированных юнита:

### Юнит 1. Реконструкция строк — `preprocess/pdf_text.py` (новый)

Чистый, детерминированный, без модели. Зависит только от `pymupdf`.

- `has_usable_text_layer(path: Path) -> bool` — True, если PDF не image-only, символов
  на страницу ≥ порога и присутствуют цифры. Отсекает PDF-сканы с пустым/мусорным слоем.
- `reconstruct_lines(path: Path) -> list[str]` — `get_text("words")` → кластеризация
  слов по координате Y с толеранцией (~3px: значение сидит на 1px ниже имени, наивное
  округление разрывает строку) → физические строки в порядке документа.
- `source_text(path: Path) -> str` — нормализованный плоский текст слоя для verbatim-стража.

Интерфейс: вход `Path`, выход — строки + метрика годности. Тестируется изолированно.

### Юнит 2. Структурирование — `llm/extract.py` + промпт в `prompts.py`

- Новый системный промпт `ANALYSIS_TEXT_SYSTEM`: «даны строки таблицы лабораторного
  бланка, разбитые по строкам документа; размести значения по колонкам (имя, значение,
  единица, референс, комментарий); КОПИРУЙ символы дословно; НЕ конвертируй единицы;
  НЕ заменяй на типичные нормы; выводи только из присутствующего текста».
- `_structure_text(lines: list[str], doc_name: str) -> list[LabResult]` — text-only
  вызов qwen3-vl (без изображения, `temp=0`) через Ollama; JSON → существующие
  `harvest_lab_rows` / `rows_from_raw` → `LabResult`. Переиспользуем готовый маппинг.

### Юнит 3. Verbatim-страж — `llm/extract.py`

- `_verbatim_guard(rows, source_text) -> (kept, rejected)` — каждое число строки
  (`value_raw` и границы референса) обязано быть подстрокой нормализованного
  `source_text`. Не прошло → строка помечается и попадает в `rejected` (не молчаливый
  дроп). Счётчик выбраковки кормит гейт фолбэка.

### Что не меняется

`normalize/` (rapidfuzz ФСЛИ), `classify` (остаётся VLM, 1000px — дёшево и работает),
persist в `pipeline/orchestrator.py`.

## Поток данных и гейт

```
run_analysis(path)
  ├─ PDF и has_usable_text_layer(path)?
  │     ├─ нет ───────────────────────────────┐
  │     └─ да:                                 │
  │         lines = reconstruct_lines(path)    │
  │         rows  = _structure_text(lines)     │
  │         rows, rej = _verbatim_guard(rows)  │
  │         strong = rows и rej/total ≤ 0.5    │
  │             ├─ strong → return rows ✅       │
  │             └─ weak ────────────────────────┤
  │                                             ▼
  └─ FALLBACK: текущий VLM-over-image (рендер → постранично → harvester)
```

- `usable` = не image-only И символов/стр ≥ порога И есть цифры.
- `weak` = 0 строк ИЛИ verbatim-страж забраковал > 50% чисел ИЛИ JSON не распарсился
  даже через `_salvage_json_objects`.
- Гейт срабатывает на документ целиком — один путь, без смешивания источников.

## Обработка ошибок

Любой сбой текстового пути ведёт к фолбэку на VLM, не к падению:
- `pymupdf` бросил / слоя нет → fallback;
- невалидный JSON модели → `_salvage_json_objects` + harvester; пусто → fallback;
- verbatim-страж забраковал большинство → weak → fallback;
- детерминизм: `temp=0`; выбранный путь логируется.

## Наблюдаемость

- `[EXTRACT_PATH] text_layer | vlm` — выбранный путь;
- `[TEXTLAYER_QUALITY] символов/стр=…, цифр=…, строк-реконструкции=…`;
- `[VERBATIM_GUARD] принято=N забраковано=M`.

## Тестирование (TDD — сначала падающий тест)

- **unit** `reconstruct_lines(sample_020)` → 21 физическая строка; строка Гемоглобина
  содержит `13.7`, `г/дл`, `11.7 - 15.5` слитно (проверка Y-толеранции).
- **unit** `has_usable_text_layer` → True для `sample_020`; False для image-only
  PDF-фикстуры.
- **unit** `_verbatim_guard` → бракует подставное `137`, пропускает `13.7`.
- **integration** `run_analysis(sample_020)` → 21 строка, Гемоглобин точный,
  идентичный результат на двух прогонах (детерминизм).
- **fallback** image-only PDF → вызывается VLM-путь (мок `_extract_once`).

## Метрики (sample_020, до → после)

| Метрика | VLM-over-image | Текстовый слой (цель) |
|---|---|---|
| Полнота | 14/21 (67%) | 21/21 (100%) |
| Точность значения | Гемоглобин `137 г/л 120-160` ❌ | `13.7 г/дл 11.7-15.5` ✅ |
| Обрыв строк | Эозинофилы `"3. "` | `3.1` целиком |
| Детерминизм | нет (`temp=0.3`) | да |
| Латентность extract | ~30–200 с (деградация до 0.3 tok/s) | <1 с + 1 малый text-вызов |
| Стоимость | 2 image-вызова (~1.2 МБ b64) | 1 text-вызов (малый промпт) |

Фактура складывается в `habr/lab-results-journal.md` (итерация 8): проблема → диагноз
(галлюцинация-нормализация на жадном декодировании) → решение (lossless-текст +
verbatim-страж) → числа.

## Вне scope (отдельные задачи, фича их не чинит)

Нормализаторные баги в `normalize/`, видимые в Doc 25: MCHC сматчен в «Средняя
концентрация гемоглобина в ретикулоците»; ложные `unit_mismatch` (Лейкоциты ждут `%`,
Тромбоциты `1/поле зр.`, Эритроциты `1/мкл`); мусорный `analyte_group`
(«Химико-микроскопические исследования» для ОАК). Также: текстовый слой для
`doctor_report`.
