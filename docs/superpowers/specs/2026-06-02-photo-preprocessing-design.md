# Дизайн: Cycle A — облегчённый препроцессинг фото/сканов

- **Дата:** 2026-06-02
- **Статус:** на согласовании
- **Область:** улучшение качества изображения перед VLM (апскейл, контраст, дескью) + подсказка
  по вводу. Часть доработки точности распознавания (после блока A+B).
- **Не входит:** интерактивная коррекция названий в боте (Cycle B), смена модели, тяжёлый
  «сканер документа» с поиском 4-угольника.

---

## 1. Контекст и доказательная база

После релиза блока A+B пользователь протестировал на реальном фото (`example.jpg`) заключения
невролога. Названия лекарств распознались с ошибками: `Глипатил` (вместо **Глиатилин**),
`Элкапар`/`Эллкар` (**Элькар**), `Тритико` (**Триттико**).

Прогон препроцессинга на этом фото (CPU, без модели) дал ключевые факты:

1. **Источник — 720×1280.** Telegram сжимает «фото» до ~1280px по длинной стороне. Текущий
   пайплайн уменьшает только то, что больше `IMAGE_MAX_LONG_SIDE=1800` — поэтому модель получила
   маленькую картинку **без апскейла**.
2. **Документ занимает ~85% кадра** (проверено маской по насыщенности). Кадрирование почти не
   добавляет разрешения; поиск 4-угольника (Canny/Otsu) на таком фото **не сработал**.
3. **Текст печатный и читаемый.** В апскейленной до 2200px + улучшенной версии все названия
   разбираются однозначно. Значит узкое место — **не кадрирование**, а (а) подача маленькой
   картинки без апскейла и (б) предел самой `qwen3-vl:8b` на плотном русском тексте.

**Вывод:** делаем *облегчённый* препроцессинг (апскейл + улучшение + безопасный дескью) и
подсказку «слать файлом» (полное разрешение камеры). Тяжёлый кроп отвергнут как бесполезный здесь.

**Ограничение проверки:** влияние на точность чтения модели проверяется только на машине
пользователя (GPU). На dev-сервере проверяется лишь корректность подготовки изображения.

## 2. Цели и критерии успеха

| # | Цель | Критерий приёмки (CPU-тест) |
|---|------|------------------------------|
| G1 | Не подавать модели маленькие картинки | Вход < target по длинной стороне **апскейлится** до target; больше — даунскейлится |
| G2 | Чётче края текста | Применяются CLAHE + unsharp; выход — валидный JPEG без падений |
| G3 | Выпрямлять реально наклонённые фото безопасно | Синтетика с наклоном >~3° выпрямляется; ровное/непонятное → полный кадр (no-op), без обрезки контента |
| G4 | Не ломать существующий поток | `classify` остаётся дешёвым (downscale-only); PDF по-прежнему даёт N изображений |
| G5 | Подсказать пользователю лучший ввод | Бот сообщает про отправку «файлом»; при низком разрешении фото — предупреждает |
| G6 | Принимать iPhone-файлы (HEIC) без 415 | Загрузка валидируется по **содержимому** (magic bytes), а не только по расширению; `.heif` в списке; HEIC с «мусорным» именем принимается и сохраняется с корректным расширением |

## 3. Архитектура

### 3.1. `preprocess/images.py` — расширение

Единый конвейер подготовки одной страницы: `resize → enhance`, плюс `deskew` для фото.

- **Resize к целевой длинной стороне (двунаправленно).** Новый параметр `IMAGE_EXTRACT_LONG_SIDE`
  (старт **2200**): для `extract` изображение приводится к этой длинной стороне — апскейл (CUBIC)
  для маленьких, даунскейл (LANCZOS) для больших. `classify` использует `IMAGE_CLASSIFY_LONG_SIDE`
  (старт 1000) **только вниз** (классификации детали не нужны).
- **Enhance.** CLAHE по L-каналу LAB (`clipLimit=2.0`, `tileGridSize=(8,8)`) + мягкий unsharp
  (`addWeighted(img,1.5, blur(σ=3),-0.5)`). Параметры подобраны замером (без гало/артефактов).
- **Deskew (best-effort, только растровые фото, не PDF).** Маска документа по низкой насыщенности
  (HSV S, Otsu-inv, морфология) → `cv2.minAreaRect` → угол нормализуется в [-45,45].
  Поворачиваем изображение, **только если** выполнены guard-условия:
  `DESKEW_MIN_ANGLE_DEG` (старт 3°) ≤ |угол| **и** доля площади маски в
  `[DESKEW_MIN_AREA, DESKEW_MAX_AREA]` (старт 0.40–0.97). Иначе — **no-op** (полный кадр).
  Кроп не делаем (документ обычно во весь кадр; агрессивный кроп рискует срезать текст).
- Порядок: `deskew (если фото) → resize → enhance → JPEG`. PDF-страницы: `resize → enhance`
  (дескью не нужен — рендер уже ровный).

> ### Почему так
> - **Альтернативы:** (A) тяжёлый сканер документа (4-угольник+перспектива) — проверено, не
>   работает на таких фото и почти бесполезен (документ во весь кадр); (B) Pillow-only
>   (autocontrast+unsharp, без дескью) — без новой тяжёлой зависимости, но CLAHE адаптивнее и
>   дескью невозможен.
> - **Выбрано:** opencv для CLAHE+дескью, апскейл маленьких входов, кроп не делаем.
> - **Компромисс:** +зависимость opencv (~60 МБ); апскейл не добавляет реальной детали (но даёт
>   модели больше патчей). Реальный эффект — только на GPU пользователя.
> - **Когда пересмотреть:** если апскейл/улучшение не повышают точность на устройстве — основной
>   рычаг переносится в Cycle B (интерактивная коррекция) и в «слать файлом».

### 3.2. Конфигурация

Новые ключи (централизованно, как остальной config): `image.extract_long_side` (2200),
`image.classify_long_side` (1000, уже есть), `image.clahe_clip` (2.0), `image.unsharp_amount`
(1.5), `image.deskew_min_angle` (3.0), `image.deskew_min_area` (0.40), `image.deskew_max_area`
(0.97). Старый `image.max_long_side` заменяется на `extract_long_side` (семантика «цель», не «потолок»).

### 3.3. Зависимости

`opencv-python-headless` (без GUI-зависимостей, подходит для сервера) и `numpy` — в рантайм.

### 3.4. Бот — подсказка по вводу (не интерактив)

- В `/help` и в ответе при приёме **фото** добавить: «📎 Для лучшего качества распознавания
  пришлите документ **файлом** (скрепка → Файл), а не фото — так сохранится полное разрешение».
- При приёме фото с длинной стороной ниже порога (`PHOTO_LOWRES_WARN`, старт 1500) — добавить
  предупреждение к ответу. Порог — в config.
- Текст — plain (бот без markdown), как принято в проекте.

### 3.5. Приём iPhone-файлов (HEIC) — валидация по содержимому

Сейчас `/upload` (api) отклоняет файл, если **расширение имени** не в списке (→ 415). iPhone
отправляет HEIC файлом часто без расширения или как `.heif` → ложный 415. Чиним так:

- Новый модуль `preprocess/formats.py`: `sniff_extension(head: bytes) -> str|None` по magic-байтам
  (PDF `%PDF`, JPEG `\xff\xd8\xff`, PNG, WEBP `RIFF…WEBP`, HEIC/HEIF — box `ftyp` + бренд
  `heic/heix/mif1/heif/…`) и `resolve_extension(filename, head, allowed)` — расширение как
  подсказка, иначе sniff.
- `/upload`: читать тело, затем `resolve_extension(...)`; 415 только если **и** расширение
  неизвестно, **и** содержимое не опознано. Сохранять файл с **корректным расширением** (важно:
  ниже по конвейеру PDF/изображение различаются по `path.suffix`).
- `config`: добавить `.heif` в `upload.allowed_extensions`.
- Бот `on_document`: если у `file_name` нет расширения — взять его из `doc.mime_type`
  (`image/heic`→`.heic`, `application/pdf`→`.pdf` и т.п.). API-снифф остаётся страховкой.
- Декодирование HEIC уже обеспечено `pillow_heif.register_heif_opener()` (вызывается при импорте
  `preprocess/images.py`, который тянется в процесс API через orchestrator).

## 5. Тестирование (CPU, без модели и сети)

- **Resize:** маленький PNG (800×600) с `extract_long_side` → длинная сторона == target (апскейл);
  большой (4000×3000) → == target (даунскейл); `classify_long_side` на маленьком → НЕ растёт.
- **Deskew:** программно повернуть синтетический «документ» (светлый прямоугольник на тёмном фоне)
  на 10° → после препроцессинга угол ≈ 0 (или хотя бы |Δ| уменьшился); ровный документ → размеры
  как при простом resize (no-op).
- **Enhance:** выход — валидный JPEG, размеры совпадают с resize; не падает на ч/б и RGB.
- **PDF:** N страниц → N изображений; дескью к PDF не применяется.
- **Guard:** изображение без явного «документа» (шум) → дескью no-op, контент не обрезан.
- Бот-подсказку проверяем юнит-логикой формирования текста (без реального Telegram).
- **Формат-снифф:** `sniff_extension` на magic-байтах (HEIC-`ftyp`, JPEG, PNG, PDF, WEBP) →
  верный ext; мусор → None. `resolve_extension`: имя с валидным ext → оно; HEIC-байты с именем
  без расширения → `.heic`. (Логика чистая, без HTTP.)

## 6. Риски и открытые вопросы

- **R1.** Дескью по маске насыщенности может ошибиться на цветных бланках/ярком фоне — поэтому
  строгие guard-условия и фолбэк на полный кадр; кроп не делаем.
- **R2.** opencv добавляет ~60 МБ в окружение. Приемлемо для локального приложения.
- **R3.** Главная неопределённость — поможет ли это модели; измеримо только на GPU пользователя.
  Митигировано тем, что Cycle B (интерактивная коррекция) даёт надёжность независимо от OCR.
